## 7. 第一份 HTML 报告怎么生成

### 7.1 最小命令

```bash
pytest --html=reports/report.html --self-contained-html
```

### 7.2 参数含义

- `--html=reports/report.html`
  指定生成报告的路径
- `--self-contained-html`
  让 HTML 报告自带依赖，更方便直接打开

### 7.3 运行前建议先建目录

```bash
mkdir -p reports
```

### 7.4 这一步带来的效果

你不再只能在终端里看结果，而是会得到一份可打开、可回看的 HTML 报告。

这对后面：

- 自己复盘
- 整理练习成果
- 对比多次运行结果

都很有帮助。

### 7.5 生成完以后到底怎么看和怎么用

很多人到这一步只会“生成”，但不会“使用”。

例如你执行：

```bash
pytest --html=reports/report.html --self-contained-html
```

那生成出来的文件就是：

```text
reports/report.html
```

你后面每次跑完测试，都可以重点看：

- 一共有多少条通过、多少条失败
- 失败的是哪条测试
- 失败堆栈停在哪一行
- 这次运行总共花了多久

也就是说，HTML 报告不是为了“存个文件”，而是为了让你在复盘时更快知道：

- 哪些测试稳定
- 哪些测试在失败
- 哪些失败需要优先处理

## 8. 为什么日志要从 Day 6 开始进入项目

如果报告告诉你“失败了”，日志负责告诉你：

- 发了什么请求
- 请求参数是什么
- 返回了什么响应

所以可以把它理解成：

- 报告：告诉你哪条测试失败
- 日志：告诉你失败时具体发生了什么

---

## 9. 最小日志应该记录什么

你现在不需要做复杂日志系统，但至少应该能记录：

1. 请求方法
2. 请求 URL
3. 请求头
4. 请求参数或请求体
5. 响应状态码
6. 响应体

如果这 6 件事你都能记录，Day 6 的日志就有价值了。

## 10. Day 6 最小日志实现示例

### 10.1 `common/logger.py`

```python
import logging


logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)s | %(message)s",
)

logger = logging.getLogger("api_test")
```

#### 放好以后到底怎么导入和使用

别的模块里通常这样导入：

```python
from common.logger import logger
```

最小使用方式：

```python
logger.info("start health test")
logger.info("response status=%s", resp.status_code)
```

这里要理解：

- `logger.py` 只负责提供统一日志对象
- 它不会自己打印业务内容
- 谁需要记日志，谁就导入它

### 10.2 在 `common/request_handler.py` 里接入日志

```python
import requests

from common.logger import logger
from config.settings import TIMEOUT


def send_request(method, url, **kwargs):
    kwargs.setdefault("timeout", TIMEOUT)

    logger.info("request start | method=%s url=%s", method, url)
    logger.info("request kwargs | %s", kwargs)

    resp = requests.request(method=method, url=url, **kwargs)

    logger.info("response status | %s", resp.status_code)
    logger.info("response text | %s", resp.text)

    return resp
```

#### 放好以后到底怎么用

真正的使用方式不是在测试文件里直接调 `logger` 记请求，而是：

1. 测试文件继续调用 `send_request()`
2. `send_request()` 内部自动记录请求和响应

例如测试里还是这样写：

```python
resp = send_request("GET", f"{BASE_URL}/health")
```

或者：

```python
resp = send_request(
    "POST",
    f"{BASE_URL}/api/v1/login",
    json={"username": "alice", "password": "123456"},
)
```

但这次和 Day 5 的区别是：

- 以前只是统一发请求
- 现在是统一发请求，同时自动记日志

也就是说，日志功能是挂在公共请求入口上的，不是让你每条测试自己去补。

### 10.3 这样做的效果

以后只要测试通过公共请求入口发请求，你就能统一看到请求和响应日志。

你不用在每个测试文件里重复写：

```python
print(resp.status_code)
print(resp.text)
```

## 11. 什么叫失败定位

失败定位不是简单地说“这条测试没过”。

真正有价值的失败定位，是你能回答：

1. 哪条测试失败了
2. 它请求了哪个接口
3. 它传了什么数据
4. 实际返回了什么
5. 是 HTTP 层失败，还是业务层失败
6. 断言具体死在哪一行

Day 6 的报告 + 日志，本质上就是在帮你回答这 6 个问题。

## 12. 故意制造一个失败用例，为什么很重要

很多人写测试时只在“都成功”的情况下看结果。

但真正能看出你是否掌握了 Day 6 的，是：

你故意让一条测试失败，看自己能不能定位。

例如：

把：

```python
assert body["data"]["status"] == "ok"
```

故意改成：

```python
assert body["data"]["status"] == "bad"
```

然后重新运行：

```bash
pytest -s -v --html=reports/report.html --self-contained-html
```

你应该能同时看到：

- 终端断言失败
- 报告里出现失败项
- 日志里能看到请求和响应

这就是 Day 6 最重要的练习之一。

## 13. 什么是测试标记 `mark`

Day 6 开始，你要学会区分不同类型的测试。

最常见的两个标记：

- `smoke`
- `regression`

### 13.1 `smoke`

冒烟测试。

特点：

- 覆盖关键主流程
- 数量少
- 跑得快

#### 这个标记放好以后怎么用

你不是打完标记就结束了，而是要配合命令去跑：

```bash
pytest -m smoke
```

这条命令的意思是：

只运行打了 `smoke` 标记的测试。

所以一条测试如果写成：

```python
@pytest.mark.smoke
def test_health():
    ...
```

那它就会被上面这条命令选中。

### 13.2 `regression`

回归测试。

特点：

- 覆盖更全面
- 数量更多
- 用于验证改动是否引入问题

#### 这个标记放好以后怎么用

对应运行命令是：

```bash
pytest -m regression
```

如果你只想跑商品链路、登录失败、更多边界场景这类更完整的测试，就可以优先跑它。

---

## 14. 标记怎么写

### 14.1 直接写在测试上

```python
import pytest


@pytest.mark.smoke
def test_health():
    ...
```

```python
import pytest


@pytest.mark.regression
def test_login():
    ...
```

### 14.2 一个测试也可以有多个标记

```python
@pytest.mark.smoke
@pytest.mark.regression
def test_profile():
    ...
```

#### 这代表什么

这代表这条测试既属于：

- 冒烟测试
- 也属于回归测试

所以它会同时被下面两种命令选中：

```bash
pytest -m smoke
pytest -m regression
```

---

## 15. 标记后怎么用

只跑冒烟测试：

```bash
pytest -m smoke
```

只跑回归测试：

```bash
pytest -m regression
```

运行非冒烟测试：

```bash
pytest -m "not smoke"
```

如果你想一边筛选，一边生成报告，也可以直接组合：

```bash
pytest -m smoke --html=reports/smoke.html --self-contained-html
```

### 15.1 这样做的效果

你不再每次都必须跑全部测试。

而是可以根据目的选：

- 快速检查主流程
- 做完整回归

## 16. `pytest.ini` 里怎么声明标记

建议你把标记写进 `pytest.ini`：

```ini
[pytest]
addopts = -v
testpaths = testcases
python_files = test_*.py
python_functions = test_*
markers =
    smoke: core flow smoke tests
    regression: broader regression tests
```

### 16.1 为什么要写进去

效果有两个：

1. 标记更规范
2. 避免 `PytestUnknownMarkWarning`

#### 这个文件放好以后怎么生效

你不需要手动导入 `pytest.ini`。

正常用法是：

1. 把它放在项目根目录
2. 直接执行 `pytest`
3. `pytest` 会自动读取里面的配置

所以：

- 标记注册写在这里
- 测试发现规则也写在这里
- 运行时自动生效

### 17.4 `testcases/test_health.py`

```python
import pytest

from common.assert_util import assert_success_response
from common.request_handler import send_request
from config.settings import BASE_URL


@pytest.mark.smoke
def test_health():
    resp = send_request("GET", f"{BASE_URL}/health")
    body = resp.json()

    assert_success_response(resp, body)
    assert body["data"]["status"] == "ok"
```

### 17.5 `testcases/test_product_flow.py`

```python
import pytest

from common.assert_util import assert_success_response
from common.request_handler import send_request
from config.settings import BASE_URL


@pytest.mark.regression
def test_product_create_query_delete(auth_headers):
    create_resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/products",
        headers=auth_headers,
        json={
            "name": "Desk Lamp",
            "category": "lighting",
            "price": 199,
        },
    )
    create_body = create_resp.json()

    assert_success_response(create_resp, create_body, status_code=201)
```

## 20. Day 6 常见错误

### 20.1 没安装 `pytest-html` 就直接加 `--html`

症状：

- 命令报参数错误

处理：

```bash
pip install pytest-html
```

### 20.2 日志只写在测试文件里

效果会很差，因为重复太多。

更好的方式是：

- 把日志尽量放到公共请求入口

### 20.3 标记写了，但没在 `pytest.ini` 注册

症状：

- 有警告

处理：

- 把 `markers` 配置补到 `pytest.ini`

### 20.4 只生成报告，不看日志

这样只能看到“结果”，看不到“过程”。

报告和日志要一起看。

### 20.5 不故意制造失败

如果你只在通过场景下看报告，其实学不到 Day 6 的精髓。
