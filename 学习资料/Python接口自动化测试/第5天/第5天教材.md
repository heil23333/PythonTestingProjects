## 4. 为什么 Day 5 必须开始重构

如果你继续沿用 Day 3 和 Day 4 的写法，代码很快会出现这些问题：

- 每个测试文件都在 `import requests`
- 每个文件都自己拼 URL
- 每个文件都自己处理响应
- 每个文件都自己写日志和断言
- 登录、请求头、公共逻辑重复很多次

当测试文件只有 2 到 3 个时，这还勉强能接受。

但一旦变成：

- 10 个测试文件
- 30 个测试场景
- 多个业务模块

你就会明显感到：

- 改一个东西要改很多处
- 找逻辑很痛苦
- 重复代码越来越多

所以 Day 5 的本质是：

从“能写测试”升级到“能组织测试代码”。

## 5. 什么叫最小框架骨架

这里说的“框架”，你不要理解成很重的企业级架构。

Day 5 的框架，先做到这几件事就够：

1. 公共请求不要在每个测试里都重复写
2. 配置集中放
3. 测试数据集中放
4. 断言尽量统一风格
5. 日志能帮助你排查问题
6. 测试文件只关心“我要测什么”

你可以把目标理解成：

“让测试文件尽量聚焦业务意图，而不是被一堆重复细节淹没。”

## 6. Day 5 推荐的最小目录结构

推荐你按这个结构理解：

```text
api_auto_test/
├── common/
│   ├── request_handler.py
│   ├── logger.py
│   └── assert_util.py
├── config/
│   ├── settings.py
│   └── env.json
├── data/
│   ├── login_cases.json
│   └── product_cases.json
├── testcases/
│   ├── test_login.py
│   └── test_product_flow.py
├── conftest.py
├── pytest.ini
└── requirements.txt
```

### 6.1 为什么这样分

#### `common/`

放公共方法。

这里放的内容通常是：

- 请求发送
- 日志
- 断言辅助

#### `config/`

放配置。

这里放的内容通常是：

- `BASE_URL`
- 账号密码
- 超时配置
- 环境配置

#### `data/`

放测试数据。

这里放的内容通常是：

- 登录测试数据
- 商品测试数据
- 参数化数据

#### `testcases/`

放真正的测试文件。

这里放的内容只应该偏“测试意图”，而不是太多公共细节。

## 7. `common` 目录到底放什么

这是 Day 5 最核心的一层。

### 7.1 `request_handler.py`

作用：

- 把最基础的 `requests.get/post/delete` 收敛起来
- 统一处理超时、公共日志等逻辑

Day 5 的目标不是封装得很复杂，而是先收敛入口。

最小示例：

```python
import requests

from config.settings import TIMEOUT


def send_request(method, url, **kwargs):
    kwargs.setdefault("timeout", TIMEOUT)
    return requests.request(method=method, url=url, **kwargs)
```

### 7.2 `logger.py`

作用：

- 统一日志格式
- 后面所有模块都能复用

最小示例：

```python
import logging


logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)s | %(message)s",
)

logger = logging.getLogger(__name__)
```

### 7.3 `assert_util.py`

作用：

- 把一些常见断言抽成小工具

最小示例：

```python
def assert_success_response(resp, body, status_code=200):
    assert resp.status_code == status_code
    assert body["code"] == 0
    assert body["message"] == "success"
```

Day 5 不需要把所有断言都抽掉，但开始有这个意识很重要。

## 8. `config` 目录到底放什么

你到 Day 4 已经知道不能每个文件都手写 `BASE_URL`。

Day 5 则要开始把配置放得更有结构。

### 8.1 `settings.py`

最适合放固定配置常量。

例如：

```python
BASE_URL = "http://127.0.0.1:5001"
USERNAME = "alice"
PASSWORD = "123456"
TIMEOUT = 5
```

### 8.2 为什么还要有 `env.json`

Day 5 可以开始让你知道“环境配置文件”这个概念。

例如：

```json
{
  "local": {
    "base_url": "http://127.0.0.1:5001",
    "timeout": 5
  }
}
```

虽然你现在先未必要用它，但你需要知道：

- 配置不一定都写在 Python 里
- 真实项目常会有环境文件

- 参数化数据文件统一放，后面很好找

## 10. `testcases` 目录应该写什么样的代码

测试文件应该尽量聚焦：

- 测什么
- 预期是什么

而不是一上来就堆很多细节。

### 10.1 不太理想的写法

```python
import requests

def test_login():
    resp = requests.post(...)
    ...
```

### 10.2 更理想的 Day 5 写法

```python
from common.request_handler import send_request
from common.assert_util import assert_success_response
from config.settings import BASE_URL


def test_health():
    resp = send_request("GET", f"{BASE_URL}/health")
    body = resp.json()

    assert_success_response(resp, body)
    assert body["data"]["status"] == "ok"
```

这样测试文件更聚焦“健康检查是否正常”。

### 11.5 `conftest.py`

```python
import pytest

from common.request_handler import send_request
from config.settings import BASE_URL, PASSWORD, USERNAME


@pytest.fixture
def reset_data():
    resp = send_request("POST", f"{BASE_URL}/api/v1/debug/reset")
    body = resp.json()

    assert resp.status_code == 200
    assert body["code"] == 0


@pytest.fixture
def login_token(reset_data):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/login",
        json={"username": USERNAME, "password": PASSWORD},
    )
    body = resp.json()

    assert resp.status_code == 200
    assert body["code"] == 0

    return body["data"]["token"]


@pytest.fixture
def auth_headers(login_token):
    return {"Authorization": f"Bearer {login_token}"}
```

### 11.6 `testcases/test_login.py`

```python
import json
from pathlib import Path

import pytest

from common.request_handler import send_request
from config.settings import BASE_URL


DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))


@pytest.mark.parametrize("case", LOGIN_CASES)
def test_login(case):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/login",
        json={
            "username": case["username"],
            "password": case["password"],
        },
    )
    body = resp.json()

    assert resp.status_code == case["expected_status"]
    assert body["code"] == case["expected_code"]
    assert body["message"] == case["expected_message"]
```

### 11.7 `testcases/test_product_flow.py`

```python
from common.assert_util import assert_success_response
from common.request_handler import send_request
from config.settings import BASE_URL


def test_product_create_query_delete(auth_headers):
    create_resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/products",
        headers=auth_headers,
        json={
            "name": "Desk Lamp",
            "category": "lighting",
            "price": 199,
        },
    )
    create_body = create_resp.json()

    assert_success_response(create_resp, create_body, status_code=201)
    product_id = create_body["data"]["id"]

    detail_resp = send_request("GET", f"{BASE_URL}/api/v1/products/{product_id}")
    detail_body = detail_resp.json()

    assert_success_response(detail_resp, detail_body)
    assert detail_body["data"]["id"] == product_id
    assert detail_body["data"]["name"] == "Desk Lamp"

    delete_resp = send_request(
        "DELETE",
        f"{BASE_URL}/api/v1/products/{product_id}",
        headers=auth_headers,
    )
    delete_body = delete_resp.json()

    assert_success_response(delete_resp, delete_body)
    assert delete_body["data"]["deleted"] is True
```

## 12. Day 5 的重构顺序应该怎么做

你不要一上来就重构全部测试。

最稳的顺序是：

1. 先建目录
2. 再把配置挪到 `config/`
3. 再把公共请求收敛到 `common/request_handler.py`
4. 再把公共 fixture 保留在 `conftest.py`
5. 最后再把测试文件搬到 `testcases/`

这个顺序很重要，因为如果你一口气全改，很容易自己把自己绕晕。

## 13. Day 5 里“封装”该封装到什么程度

这是最容易走偏的地方。

### 13.1 不要过度封装

Day 5 不建议你上来就写：

- 很复杂的基类
- 很厚的抽象层
- 一大堆装饰器
- 难懂的统一入口

因为你现在最重要的是可读性，不是炫技。

### 13.2 Day 5 的封装只要做到

- 请求统一入口
- 配置统一入口
- 公共断言适度抽取
- 公共 fixture 集中管理

就够了。

## 14. 为什么日志要在 Day 5 开始进入项目结构

前面你可能靠 `print` 也能排查。

但当你开始有多个模块后，日志就很重要了。

最起码你要能看到：

- 请求方法
- 请求 URL
- 请求参数
- 响应状态码
- 响应内容

Day 5 不要求你做完整日志系统，但你要知道：

- 日志应该开始进入公共模块
- 不应一直散落在测试代码里

## 15. Day 5 里的断言为什么还要继续整理

因为现在你已经开始有多个测试文件。

如果每个地方都随手写：

```python
assert resp.status_code == 200
assert body["code"] == 0
assert body["message"] == "success"
```

重复会越来越多。

所以适度抽成：

```python
assert_success_response(resp, body)
```

是合理的。

但要记住：

- 通用断言可以抽
- 业务特有断言仍然放在测试里

例如：

```python
assert body["data"]["username"] == "alice"
```

这种就不一定要抽。

## 16. Day 5 常见错误

### 16.1 目录建了，但测试仍然全写在根目录

这属于“看起来有结构，实际上没迁移”。

### 16.2 `common` 模块越封越厚

Day 5 不追求一口气搭成框架，只求先清晰。

### 16.3 公共方法抽太早太多

有些逻辑还没稳定，先别急着抽象。

### 16.4 测试文件里仍然全是重复请求代码

说明你的公共请求入口还没真正用起来。

### 16.5 配置和测试数据仍然混在一起

这说明 Day 4 到 Day 5 的迁移还没完成。
