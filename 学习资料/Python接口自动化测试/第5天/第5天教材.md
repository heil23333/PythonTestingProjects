## 4. 为什么 Day 5 必须开始重构

如果你继续沿用 Day 3 和 Day 4 的写法，代码很快会出现这些问题：

- 每个测试文件都在 `import requests`
- 每个文件都自己拼 URL
- 每个文件都自己处理响应
- 每个文件都自己写日志和断言
- 登录、请求头、公共逻辑重复很多次

当测试文件只有 2 到 3 个时，这还勉强能接受。

但一旦变成：

- 10 个测试文件
- 30 个测试场景
- 多个业务模块

你就会明显感到：

- 改一个东西要改很多处
- 找逻辑很痛苦
- 重复代码越来越多

所以 Day 5 的本质是：

从“能写测试”升级到“能组织测试代码”。

## 5. 什么叫最小框架骨架

这里说的“框架”，你不要理解成很重的企业级架构。

Day 5 的框架，先做到这几件事就够：

1. 公共请求不要在每个测试里都重复写
2. 配置集中放
3. 测试数据集中放
4. 断言尽量统一风格
5. 日志能帮助你排查问题
6. 测试文件只关心“我要测什么”

你可以把目标理解成：

“让测试文件尽量聚焦业务意图，而不是被一堆重复细节淹没。”

## 6. Day 5 推荐的最小目录结构

推荐你按这个结构理解：

```text
api_auto_test/
├── common/
│   ├── request_handler.py
│   ├── logger.py
│   └── assert_util.py
├── config/
│   ├── settings.py
│   └── env.json
├── data/
│   ├── login_cases.json
│   └── product_cases.json
├── testcases/
│   ├── test_login.py
│   └── test_product_flow.py
├── conftest.py
├── pytest.ini
└── requirements.txt
```

### 6.1 为什么这样分

#### `common/`

放公共方法。

这里放的内容通常是：

- 请求发送
- 日志
- 断言辅助

#### `config/`

放配置。

这里放的内容通常是：

- `BASE_URL`
- 账号密码
- 超时配置
- 环境配置

#### `data/`

放测试数据。

这里放的内容通常是：

- 登录测试数据
- 商品测试数据
- 参数化数据

#### `testcases/`

放真正的测试文件。

这里放的内容只应该偏“测试意图”，而不是太多公共细节。

### 6.2 这样分层之后的直接效果

如果你只是看目录树，很容易觉得“只是把文件换了个位置”。

但真正的效果不是位置变化，而是维护方式变化。

#### 效果 1：改配置时，改动范围明显变小

例如你以后把：

```python
BASE_URL = "http://127.0.0.1:5001"
```

改成：

```python
BASE_URL = "http://127.0.0.1:5002"
```

如果你没有分层，可能要改很多测试文件。

如果你已经把配置集中到 `config/`，通常只改一处。

#### 效果 2：测试文件更容易读

分层前你看到一个测试文件，里面可能同时混着：

- 请求发送细节
- 配置常量
- 登录逻辑
- 断言逻辑
- 业务步骤

分层后，测试文件会更接近：

- 我要测什么
- 我期待什么

这样你自己回头看代码也更容易。

#### 效果 3：重复代码开始减少

例如：

- 超时设置
- 请求入口
- 成功响应断言

这些如果不抽出去，后面几乎会在每个文件重复出现。

#### 效果 4：后面新增测试更快

因为你不需要每次新增一个测试都重新拼完整请求和断言框架。

你只要：

1. 取配置
2. 调公共请求
3. 写业务断言

就能继续往下加。

## 7. `config` 目录到底放什么

你到 Day 4 已经知道不能每个文件都手写 `BASE_URL`。

Day 5 则要开始把配置放得更有结构。

而且这一步必须先做。

原因很直接：

- `common/request_handler.py` 里的默认超时要用 `TIMEOUT`
- `conftest.py` 里的登录和重置要用 `BASE_URL`、账号密码
- 测试文件里的 URL 拼接也要用 `BASE_URL`

所以 Day 5 真正的搭建顺序不是“随便先放哪个目录都行”，而是：

先把 `config/` 定下来，再让其他模块依赖它。

### 7.1 `settings.py`

最适合放固定配置常量。

例如：

```python
BASE_URL = "http://127.0.0.1:5001"
USERNAME = "alice"
PASSWORD = "123456"
TIMEOUT = 5
```

#### 这样放的效果

你以后只要碰到这些场景，就能明显感觉到配置集中管理的好处：

1. 改端口时不用满项目搜索
2. 换账号时不用逐个测试文件修改
3. 默认超时统一管理

这一步的实际效果是：

把“环境信息”从“测试逻辑”里剥离出来。

#### 放好以后到底怎么导入和使用

最常见的导入方式是：

```python
from config.settings import BASE_URL, TIMEOUT
```

如果你要导入账号密码，也可以这样写：

```python
from config.settings import BASE_URL, USERNAME, PASSWORD, TIMEOUT
```

最典型的使用场景有 3 个：

1. 在测试文件里拼 URL

```python
from config.settings import BASE_URL

resp = send_request("GET", f"{BASE_URL}/health")
```

2. 在公共请求入口里拿默认超时

```python
from config.settings import TIMEOUT
```

3. 在 `conftest.py` 里拿登录账号

```python
from config.settings import BASE_URL, USERNAME, PASSWORD
```

也就是说，`config/settings.py` 不是“放在那里备用”，而是后面很多模块都会直接导入它。

### 7.2 为什么还要有 `env.json`

Day 5 可以开始让你知道“环境配置文件”这个概念。

例如：

```json
{
  "local": {
    "base_url": "http://127.0.0.1:5001",
    "timeout": 5
  }
}
```

虽然你现在先未必要用它，但你需要知道：

- 配置不一定都写在 Python 里
- 真实项目常会有环境文件

#### 这样放的效果

这一步对你当前练习项目也许还不明显，但对真实项目很重要。

因为你迟早会遇到：

- 本地环境
- 测试环境
- 预发布环境

如果没有环境文件，切换环境会非常痛苦。

所以 Day 5 先让你知道“环境配置应该可切换”，这是为后面打基础。

#### 当前阶段怎么理解它的用法

Day 5 你可以先把它理解成“环境数据仓库”，当前不一定马上接进代码里。

也就是说：

- `settings.py` 现在先直接给测试项目使用
- `env.json` 先让你知道以后可以把不同环境的信息放进去

如果后面你想读它，常见写法会像这样：

```python
import json
from pathlib import Path

ENV_FILE = Path(__file__).resolve().parent / "env.json"
env_data = json.loads(ENV_FILE.read_text(encoding="utf-8"))
local_base_url = env_data["local"]["base_url"]
```

Day 5 不要求你马上用上这段代码，但至少要知道：

这个文件不是“装饰品”，它的未来用途就是承载多环境配置。

## 8. `common` 目录到底放什么

这是 Day 5 最核心的一层，但它不是最先落地的一层。

正确顺序是：

1. 先把 `config/settings.py` 放好
2. 再让 `common/request_handler.py` 去依赖 `TIMEOUT`
3. 再让测试文件和 `conftest.py` 去依赖公共请求入口

也就是说：

`common/` 很重要，但它建立在 `config/` 已经稳定的前提上。

### l8.1 `request_handler.py`

作用：

- 把最基础的 `requests.get/post/delete` 收敛起来
- 统一处理超时、公共日志等逻辑

Day 5 的目标不是封装得很复杂，而是先收敛入口。

最小示例：

```python
import requests

from config.settings import TIMEOUT


def send_request(method, url, **kwargs):
    kwargs.setdefault("timeout", TIMEOUT)
    return requests.request(method=method, url=url, **kwargs)
```

#### 为什么这里一定要先有 `config`

因为这段代码里已经出现了：

```python
from config.settings import TIMEOUT
```

如果你还没有先把 `config/settings.py` 建好，这个公共请求入口就没有稳定依赖。

所以它不是孤立存在的，而是建立在配置层之上的。

#### 这样放的效果

把请求统一收敛到这里之后，你会立刻得到这些收益：

1. 默认超时不需要每个文件重复写
2. 如果以后要统一加日志、统一加 headers、统一做异常处理，只改一个地方
3. 测试文件不再到处直接调用 `requests.get/post/delete`

也就是说，它的效果不是“封装一下更优雅”，而是：

后面你一旦想改请求层行为，不需要挨个测试文件去搜。

#### 放好以后到底怎么调用

测试文件里先这样导入：

```python
from common.request_handler import send_request
```

然后最常见的调用方式有这些：

1. 发一个最简单的 GET 请求

```python
resp = send_request("GET", f"{BASE_URL}/health")
```

2. 发一个带 JSON 请求体的 POST 请求

```python
resp = send_request(
    "POST",
    f"{BASE_URL}/api/v1/login",
    json={"username": "alice", "password": "123456"},
)
```

3. 发一个带请求头的 GET 请求

```python
resp = send_request(
    "GET",
    f"{BASE_URL}/api/v1/profile",
    headers={"Authorization": f"Bearer {token}"},
)
```

4. 发一个带查询参数的 GET 请求

```python
resp = send_request(
    "GET",
    f"{BASE_URL}/api/v1/products",
    params={"page": 1, "page_size": 10},
)
```

5. 发一个 DELETE 请求

```python
resp = send_request(
    "DELETE",
    f"{BASE_URL}/api/v1/products/{product_id}",
    headers=auth_headers,
)
```

6. 临时覆盖默认超时

```python
resp = send_request("GET", f"{BASE_URL}/health", timeout=10)
```

这里要理解清楚：

- `send_request()` 不是只能发一种请求
- 它只是把 `requests.request()` 包了一层
- `json`、`headers`、`params`、`timeout` 这些还是照常传

所以你后面写测试时，几乎可以把它理解成：

“统一版的 `requests.get/post/delete` 入口”

### 8.2 `logger.py`

作用：

- 统一日志格式
- 后面所有模块都能复用

最小示例：

```python
import logging


logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)s | %(message)s",
)

logger = logging.getLogger(__name__)
```

#### 这样放的效果

如果日志一直散落在测试文件里，会出现两个问题：

1. 每个文件日志格式不一致
2. 以后想统一调整日志级别很麻烦

把日志入口收进公共模块后，你得到的效果是：

- 输出风格统一
- 后面可以逐步升级日志能力
- 测试文件里不需要反复写初始化逻辑

#### 放好以后到底怎么用

别的模块里通常这样导入：

```python
from common.logger import logger
```

最简单的使用方式：

```python
logger.info("start login request")
logger.info("response status=%s", resp.status_code)
```

放到 `request_handler.py` 里时，常见写法会像这样：

```python
logger.info("request start | method=%s url=%s", method, url)
logger.info("request kwargs | %s", kwargs)
logger.info("response status | %s", resp.status_code)
logger.info("response text | %s", resp.text)
```

也就是说：

- 不是每个测试文件自己初始化日志
- 而是统一导入同一个 `logger`
- 谁需要记录日志，谁就导入它

### 8.3 `assert_util.py`

作用：

- 把一些常见断言抽成小工具

最小示例：

```python
def assert_success_response(resp, body, status_code=200):
    assert resp.status_code == status_code
    assert body["code"] == 0
    assert body["message"] == "success"
```

Day 5 不需要把所有断言都抽掉，但开始有这个意识很重要。

#### 这样放的效果

把通用断言抽出来之后，最大的效果不是“代码变少几行”，而是：

1. 断言风格更统一
2. 以后如果成功响应规则变化，只需要改一个方法
3. 测试文件更聚焦业务字段，而不是重复写相同断言

例如：

原来每个文件都写：

```python
assert resp.status_code == 200
assert body["code"] == 0
assert body["message"] == "success"
```

后面统一成：

```python
assert_success_response(resp, body)
```

这样测试文件就更容易扫读。

#### 放好以后到底怎么调用

测试文件里这样导入：

```python
from common.assert_util import assert_success_response
```

最常见的调用方式有两种。

1. 默认按成功 `200` 断言

```python
resp = send_request("GET", f"{BASE_URL}/health")
body = resp.json()

assert_success_response(resp, body)
```

2. 如果接口成功状态码不是 `200`，就显式传入

```python
create_resp = send_request(
    "POST",
    f"{BASE_URL}/api/v1/products",
    headers=auth_headers,
    json={"name": "Desk Lamp", "category": "lighting", "price": 199},
)
create_body = create_resp.json()

assert_success_response(create_resp, create_body, status_code=201)
```

它的边界也要搞清楚：

- 通用成功断言可以抽到这里
- 业务特有断言仍然要写在测试文件里

例如：

```python
assert create_body["data"]["name"] == "Desk Lamp"
```

这种就不应该硬塞进公共断言里。

## 9. `data` 目录到底放什么

这里就是放参数化数据和业务测试数据。

### 9.1 `login_cases.json`

最典型。

```json
[
  {
    "username": "alice",
    "password": "123456",
    "expected_status": 200,
    "expected_code": 0,
    "expected_message": "success"
  },
  {
    "username": "alice",
    "password": "wrong",
    "expected_status": 401,
    "expected_code": 10014,
    "expected_message": "password error"
  }
]
```

#### 放好以后到底怎么读

测试文件里通常这样写：

```python
import json
from pathlib import Path

DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))
```

然后给参数化用：

```python
@pytest.mark.parametrize("case", LOGIN_CASES)
def test_login(case):
    ...
```

这里的实际用法是：

- `DATA_FILE` 负责找到数据文件
- `LOGIN_CASES` 负责把 JSON 数组读成 Python 列表
- `case` 负责在每次测试时拿到其中一条数据

### 9.2 `product_cases.json`

可以放商品测试数据，例如：

```json
[
  {
    "name": "Desk Lamp",
    "category": "lighting",
    "price": 199
  }
]
```

#### 放好以后到底怎么读和用

如果你把商品数据放进这个文件，读取方式和登录数据一样：

```python
import json
from pathlib import Path

DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "product_cases.json"
PRODUCT_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))
```

如果这个文件里只有一组数据，你可以直接按索引取：

```python
case = PRODUCT_CASES[0]
```

如果有多组数据，也可以继续参数化：

```python
@pytest.mark.parametrize("case", PRODUCT_CASES)
def test_create_product(case, auth_headers):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/products",
        headers=auth_headers,
        json=case,
    )
```

### 9.3 为什么 Day 5 要强调数据目录

因为当测试越来越多时：

- 数据和测试逻辑混在一起会很难维护
- 参数化数据文件统一放，后面很好找

#### 这样放的效果

把数据收进 `data/` 后，你会立刻得到两个实际好处：

1. 测试函数更短
2. 后面改测试数据时，不容易误伤测试逻辑

也就是说：

- 你想改“测什么数据”，去 `data/`
- 你想改“怎么测”，去 `testcases/`

职责开始清晰。

## 10. `testcases` 目录应该写什么样的代码

测试文件应该尽量聚焦：

- 测什么
- 预期是什么

而不是一上来就堆很多细节。

### 10.1 不太理想的写法

```python
import requests

def test_login():
    resp = requests.post(...)
    ...
```

### 10.2 更理想的 Day 5 写法

```python
from common.request_handler import send_request
from common.assert_util import assert_success_response
from config.settings import BASE_URL


def test_health():
    resp = send_request("GET", f"{BASE_URL}/health")
    body = resp.json()

    assert_success_response(resp, body)
    assert body["data"]["status"] == "ok"
```

这样测试文件更聚焦“健康检查是否正常”。

### 10.3 这样放进 `testcases/` 的效果

测试文件放进 `testcases/` 目录，不只是为了整齐。

它的直接效果是：

1. 你能一眼找到所有测试入口
2. 后面 `pytest.ini` 可以只扫这个目录
3. 测试文件不会和配置、数据、公共模块混在一起

这一步对“找文件”和“控制测试范围”都很有帮助。

#### 这个文件放好以后怎么用

你平时不需要手动导入它。

它的用法是：

- 你把它放在项目根目录
- 然后直接运行 `pytest`
- `pytest` 会自动读取里面的配置

例如如果你写了：

```ini
[pytest]
testpaths = testcases
```

那么你在项目根目录执行：

```bash
pytest
```

它就会优先去 `testcases/` 找测试，而不是整目录乱扫。

这就是为什么 `pytest.ini` 看起来像“只是一个配置文件”，但实际很影响你的日常使用体验。

## 11.10 重构前后对比

为了让你更直观地理解 Day 5 的意义，可以直接看下面这个对比。

### 重构前

一个测试文件里可能同时有：

```python
import requests

BASE_URL = "http://127.0.0.1:5001"


def test_health():
    resp = requests.get(f"{BASE_URL}/health", timeout=5)
    body = resp.json()

    assert resp.status_code == 200
    assert body["code"] == 0
    assert body["message"] == "success"
    assert body["data"]["status"] == "ok"
```

这种写法的问题不是“不能跑”，而是：

- `BASE_URL` 重复
- `timeout` 重复
- 通用断言重复
- 公共逻辑和业务逻辑混在一起

### 重构后

```python
from common.assert_util import assert_success_response
from common.request_handler import send_request
from config.settings import BASE_URL


def test_health():
    resp = send_request("GET", f"{BASE_URL}/health")
    body = resp.json()

    assert_success_response(resp, body)
    assert body["data"]["status"] == "ok"
```

这种写法的效果是：

- 业务意图更突出
- 重复代码更少
- 改公共逻辑的成本更低

这就是 Day 5 真正想让你获得的能力。

## 11.11 这些文件放好以后，到底怎么用

这是 Day 5 最容易让人卡住的地方。

很多人看到目录结构会觉得：

- 我知道该放哪里了
- 但我不知道之后写测试时要怎么调用这些文件

所以这里直接按“你平时写测试的动作顺序”说明。

### 第一步：测试文件先导入配置

例如你在 `testcases/test_health.py` 里，先拿配置：

```python
from config.settings import BASE_URL
```

这一步的实际用法就是：

- 不再自己手写 `BASE_URL`
- 以后地址改了，测试文件不用动

### 第二步：测试文件再导入公共请求方法

```python
from common.request_handler import send_request
```

这一步的实际用法就是：

- 不再直接到处写 `requests.get/post/delete`
- 测试文件统一从公共入口发请求

也就是说，原来你写：

```python
resp = requests.get(f"{BASE_URL}/health", timeout=5)
```

现在会变成：

```python
resp = send_request("GET", f"{BASE_URL}/health")
```

效果是：

- 超时不需要每个文件重复写
- 后面如果请求层逻辑变了，只改公共模块

### 第三步：测试文件导入公共断言

```python
from common.assert_util import assert_success_response
```

这一步的实际用法就是：

- 不再每个文件都重复写成功断言

原来你写：

```python
assert resp.status_code == 200
assert body["code"] == 0
assert body["message"] == "success"
```

现在会变成：

```python
assert_success_response(resp, body)
```

### 第四步：测试数据文件通过路径读取

如果你有 `data/login_cases.json`，测试里通常这样用：

```python
import json
from pathlib import Path

DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))
```

这一步的实际用法就是：

- 测试数据不写死在测试函数里
- 以后你想加场景，只改数据文件

### 第五步：fixture 继续从 `conftest.py` 自动注入

你不需要在测试文件里这样写：

```python
from conftest import auth_headers
```

不要这么写。

正确用法是：

只要 `conftest.py` 在项目里，测试函数直接写参数名就可以：

```python
def test_product_flow(auth_headers):
    ...
```

这一步的实际效果是：

- 测试函数直接拿到前置数据
- 但测试文件里不需要自己处理登录细节

### 第六步：pytest 只跑 `testcases/` 目录

如果 `pytest.ini` 里写了：

```ini
[pytest]
addopts = -v
testpaths = testcases
python_files = test_*.py
python_functions = test_*
```

那你以后运行：

```bash
pytest
```

它就会优先去 `testcases/` 里找测试。

这一步的实际效果是：

- 配置文件、公共模块、数据文件不会被当成测试扫
- 项目入口更清晰

## 11.13 一个完整的“实际怎么用”例子

假设你已经有这些文件：

- `config/settings.py`
- `common/request_handler.py`
- `common/assert_util.py`
- `conftest.py`
- `data/login_cases.json`
- `testcases/test_login.py`

那你平时写测试时，真正的使用方式会是这样：

```python
import json
from pathlib import Path

import pytest

from common.request_handler import send_request
from config.settings import BASE_URL


DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))


@pytest.mark.parametrize("case", LOGIN_CASES)
def test_login(case):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/login",
        json={
            "username": case["username"],
            "password": case["password"],
        },
    )
    body = resp.json()

    assert resp.status_code == case["expected_status"]
    assert body["code"] == case["expected_code"]
    assert body["message"] == case["expected_message"]
```

这段代码里每个部分的“使用价值”分别是：

- `config.settings`
  负责拿配置
- `common.request_handler`
  负责统一发请求
- `data/login_cases.json`
  负责提供测试数据
- `pytest.mark.parametrize`
  负责让同一个测试跑多组数据

这就是 Day 5 真正的使用方式。

## 12. Day 5 的重构顺序应该怎么做

你不要一上来就重构全部测试。

最稳的顺序是：

1. 先建目录
2. 再把配置挪到 `config/`
3. 再把公共请求收敛到 `common/request_handler.py`
4. 再把公共 fixture 保留在 `conftest.py`
5. 最后再把测试文件搬到 `testcases/`

这个顺序很重要，因为如果你一口气全改，很容易自己把自己绕晕。

## 13. Day 5 里“封装”该封装到什么程度

这是最容易走偏的地方。

### 13.1 不要过度封装

Day 5 不建议你上来就写：

- 很复杂的基类
- 很厚的抽象层
- 一大堆装饰器
- 难懂的统一入口

因为你现在最重要的是可读性，不是炫技。

### 13.2 Day 5 的封装只要做到

- 请求统一入口
- 配置统一入口
- 公共断言适度抽取
- 公共 fixture 集中管理

## 15. Day 5 里的断言为什么还要继续整理

因为现在你已经开始有多个测试文件。

如果每个地方都随手写：

```python
assert resp.status_code == 200
assert body["code"] == 0
assert body["message"] == "success"
```

重复会越来越多。

所以适度抽成：

```python
assert_success_response(resp, body)
```

是合理的。

但要记住：

- 通用断言可以抽
- 业务特有断言仍然放在测试里

例如：

```python
assert body["data"]["username"] == "alice"
```

这种就不一定要抽。
