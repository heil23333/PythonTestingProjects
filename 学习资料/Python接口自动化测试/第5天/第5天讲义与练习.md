## 4. 讲义模块一：先建目录

今天先把目录建出来，再迁移代码。

推荐结构：

```text
api_auto_test/
├── common/
├── config/
├── data/
├── testcases/
├── conftest.py
├── pytest.ini
└── requirements.txt
```

### 你要理解

- `common` 放公共方法
- `config` 放配置
- `data` 放测试数据
- `testcases` 放测试文件

### 这样做的效果

不是“目录更好看”，而是：

- 你以后找测试文件更快
- 找配置更快
- 找测试数据更快
- 修改公共逻辑时不用到处改

### 4.1 建完目录以后怎么开始用

今天不是建完目录就结束。

建完目录以后，你应该立刻开始做下面这几件事：

1. 把配置文件放进 `config/`
2. 把公共请求方法放进 `common/`
3. 把测试数据放进 `data/`
4. 把测试文件挪进 `testcases/`
5. 跑一次 `pytest`

也就是说：

目录不是给你“看着舒服”的，是为了支撑后面的使用流程。

这里还有一个顺序要求要记住：

今天迁移时，最先真正落地的不是 `common/`，而是 `config/`。

因为后面这些地方都要先依赖配置：

- `common/request_handler.py` 要用 `TIMEOUT`
- `conftest.py` 要用 `BASE_URL`、账号密码
- 测试文件也要用 `BASE_URL`

所以你如果顺序搞反，就会出现“公共请求入口已经写了，但依赖的配置还没放好”的情况。

### 随堂检查

1. 为什么不建议所有东西都继续堆在根目录？

### 参考答案

1. 因为测试一多，配置、数据、公共逻辑和测试文件会混在一起，后面很难维护。

## 5. 讲义模块二：先迁配置

先建：

```text
config/settings.py
```

内容：

```python
BASE_URL = "http://127.0.0.1:5001"
USERNAME = "alice"
PASSWORD = "123456"
TIMEOUT = 5
```

### 你要理解

- 先把 `BASE_URL`、账号密码从测试文件里收走

### 这样做的效果

以后你改端口、改账号、改超时，只需要改一处。

### 5.1 放好以后怎么用

测试文件里这样导入：

```python
from config.settings import BASE_URL, TIMEOUT
```

然后直接拿来拼 URL 或传超时：

```python
resp = send_request("GET", f"{BASE_URL}/health", timeout=TIMEOUT)
```

### 5.2 为什么 Day 5 必须先迁配置

不是因为 `config/` 这个目录更基础、更“标准”，而是因为它是后面代码的依赖起点。

例如你马上下一步要写的：

```python
from config.settings import TIMEOUT
```

如果 `settings.py` 还没有先建好，这一步就无从谈起。

所以 Day 5 的正确搭建思路是：

先让配置稳定，再让公共请求、fixture、测试文件都去依赖它。

## 6. 讲义模块三：再迁公共请求

新建：

```text
common/request_handler.py
```

内容：

```python
import requests

from config.settings import TIMEOUT


def send_request(method, url, **kwargs):
    kwargs.setdefault("timeout", TIMEOUT)
    return requests.request(method=method, url=url, **kwargs)
```

### 你要理解

- 后面测试不再直接到处写 `requests.get/post/delete`
- 先统一走一个入口
- 这个入口之所以能工作，是因为它已经依赖了前一步准备好的 `config.settings`

### 这样做的效果

以后你想统一加：

- 超时
- 日志
- 异常处理

只需要改一个文件，而不是改很多测试文件。

### 6.1 放好以后怎么用

测试文件里这样导入：

```python
from common.request_handler import send_request
```

然后原来这些：

```python
requests.get(...)
requests.post(...)
requests.delete(...)
```

逐步改成：

```python
send_request("GET", ...)
send_request("POST", ...)
send_request("DELETE", ...)
```

### 6.2 `send_request` 最常见的实际调用方式

这一步你不能只知道“要导入 `send_request`”，还要知道导入以后到底怎么写。

最常见的 5 种写法如下。

1. 健康检查

```python
resp = send_request("GET", f"{BASE_URL}/health")
```

2. 登录

```python
resp = send_request(
    "POST",
    f"{BASE_URL}/api/v1/login",
    json={"username": "alice", "password": "123456"},
)
```

3. 查个人资料

```python
resp = send_request(
    "GET",
    f"{BASE_URL}/api/v1/profile",
    headers={"Authorization": f"Bearer {token}"},
)
```

4. 查商品列表

```python
resp = send_request(
    "GET",
    f"{BASE_URL}/api/v1/products",
    params={"page": 1, "page_size": 10},
)
```

5. 删除商品

```python
resp = send_request(
    "DELETE",
    f"{BASE_URL}/api/v1/products/{product_id}",
    headers=auth_headers,
)
```

你要记住：

- 请求方法还是写字符串：`GET`、`POST`、`DELETE`
- `json`、`headers`、`params` 的传法不变
- 只是把原来分散的 `requests.get/post/delete` 收敛到一个入口

### 随堂检查

1. 为什么这里用 `kwargs.setdefault("timeout", TIMEOUT)`？

### 参考答案

1. 因为这样默认有超时配置，但需要时也可以单独覆盖。

## 7. 讲义模块四：再迁公共断言

新建：

```text
common/assert_util.py
```

内容：

```python
def assert_success_response(resp, body, status_code=200):
    assert resp.status_code == status_code
    assert body["code"] == 0
    assert body["message"] == "success"
```

### 你要理解

- 通用断言抽出来后，测试文件会更干净

### 这样做的效果

以后你改“成功响应的通用规则”时，只改断言工具方法，不用每个测试都找一遍。

### 7.1 放好以后怎么用

测试文件里导入：

```python
from common.assert_util import assert_success_response
```

然后把通用成功断言写成：

```python
assert_success_response(resp, body)
```

业务特有断言继续写在测试里：

```python
assert body["data"]["name"] == "Desk Lamp"
```

### 7.2 `assert_success_response` 到底什么时候用

这类公共断言适合放在“成功响应规则一致”的地方。

例如：

```python
resp = send_request("GET", f"{BASE_URL}/health")
body = resp.json()

assert_success_response(resp, body)
```

如果接口成功状态码是 `201`，就显式传进去：

```python
create_resp = send_request(
    "POST",
    f"{BASE_URL}/api/v1/products",
    headers=auth_headers,
    json={"name": "Desk Lamp", "category": "lighting", "price": 199},
)
create_body = create_resp.json()

assert_success_response(create_resp, create_body, status_code=201)
```

但下面这种业务断言，仍然应该保留在测试文件里：

```python
assert create_body["data"]["name"] == "Desk Lamp"
```

## 8. 讲义模块五：整理 conftest.py

新建或迁移为：

```python
import pytest

from common.request_handler import send_request
from config.settings import BASE_URL, PASSWORD, USERNAME


@pytest.fixture
def reset_data():
    resp = send_request("POST", f"{BASE_URL}/api/v1/debug/reset")
    body = resp.json()

    assert resp.status_code == 200
    assert body["code"] == 0


@pytest.fixture
def login_token(reset_data):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/login",
        json={"username": USERNAME, "password": PASSWORD},
    )
    body = resp.json()

    assert resp.status_code == 200
    assert body["code"] == 0

    return body["data"]["token"]


@pytest.fixture
def auth_headers(login_token):
    return {"Authorization": f"Bearer {login_token}"}
```

### 你要理解

- Day 5 不是删掉 fixture，而是让 fixture 和公共请求一起工作

### 这样做的效果

fixture 继续负责前置步骤，但请求细节已经被统一管理，职责开始更清楚。

### 8.1 放好以后怎么用

测试文件里不要手动导入 `conftest.py`。

只要测试函数参数里写 fixture 名字就行：

```python
def test_product_flow(auth_headers):
    ...
```

这就是使用方式。

这里最容易误解的点是：

- 你不会手动写 `from conftest import auth_headers`
- 你只需要把 fixture 名字写进测试函数参数
- `pytest` 会自动帮你注入

## 9. 讲义模块六：迁移登录测试

新建：

```text
testcases/test_login.py
```

内容：

```python
import json
from pathlib import Path

import pytest

from common.request_handler import send_request
from config.settings import BASE_URL


DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))


@pytest.mark.parametrize("case", LOGIN_CASES)
def test_login(case):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/login",
        json={
            "username": case["username"],
            "password": case["password"],
        },
    )
    body = resp.json()

    assert resp.status_code == case["expected_status"]
    assert body["code"] == case["expected_code"]
    assert body["message"] == case["expected_message"]
```

### 你要理解

- Day 5 的测试文件开始更像“业务测试”
- 公共细节已经在慢慢下沉

### 9.1 放好以后怎么用

这里的关键不是“把文件挪进去”，而是：

测试文件里以后主要只写：

1. 导入配置
2. 导入公共请求方法
3. 导入公共断言
4. 写业务步骤和业务断言

### 这样做的效果

你以后打开测试文件时，更容易一眼看懂“它到底在测什么”。

### 9.2 数据文件在测试里到底怎么用

如果你已经有：

```text
data/login_cases.json
```

测试文件里最常见的读取方式就是：

```python
import json
from pathlib import Path

DATA_FILE = Path(__file__).resolve().parent.parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))
```

然后再这样用：

```python
@pytest.mark.parametrize("case", LOGIN_CASES)
def test_login(case):
    resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/login",
        json={
            "username": case["username"],
            "password": case["password"],
        },
    )
```

这里的使用顺序是：

1. 先定位数据文件
2. 再读成 Python 列表
3. 再把每条数据交给参数化测试

## 10. 讲义模块七：迁移商品链路测试

新建：

```text
testcases/test_product_flow.py
```

内容：

```python
from common.assert_util import assert_success_response
from common.request_handler import send_request
from config.settings import BASE_URL


def test_product_create_query_delete(auth_headers):
    create_resp = send_request(
        "POST",
        f"{BASE_URL}/api/v1/products",
        headers=auth_headers,
        json={
            "name": "Desk Lamp",
            "category": "lighting",
            "price": 199,
        },
    )
    create_body = create_resp.json()

    assert_success_response(create_resp, create_body, status_code=201)
    product_id = create_body["data"]["id"]

    detail_resp = send_request("GET", f"{BASE_URL}/api/v1/products/{product_id}")
    detail_body = detail_resp.json()

    assert_success_response(detail_resp, detail_body)
    assert detail_body["data"]["id"] == product_id
    assert detail_body["data"]["name"] == "Desk Lamp"

    delete_resp = send_request(
        "DELETE",
        f"{BASE_URL}/api/v1/products/{product_id}",
        headers=auth_headers,
    )
    delete_body = delete_resp.json()

    assert_success_response(delete_resp, delete_body)
    assert delete_body["data"]["deleted"] is True
```

### 你要理解

- 这是 Day 5 的核心迁移成果
- 测试文件已经明显比 Day 3 更整洁了

### 10.1 放好以后怎么用

这种链路测试迁移到 `testcases/` 后，正常使用方式就是：

```bash
pytest
pytest -k product
pytest testcases/test_product_flow.py
```

也就是说，它已经变成项目结构里可直接执行的一部分，而不是散落在某个练习脚本目录里的单文件。

这里也要注意：

- `testcases/` 里的测试文件不是孤立脚本
- 它们会一起依赖 `config/`、`common/`、`conftest.py`
- 所以你平时应该在项目根目录运行 `pytest`

### 这样做的效果

这类链路测试以后还能继续变长，但不会把公共配置、请求入口、断言风格一起搅乱。
