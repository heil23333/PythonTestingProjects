## 4. Day 7 为什么很关键

很多人学接口自动化时，前面几天学得都还行，但最后停在这里：

- 会写零散脚本
- 会跑几个测试
- 会用 `pytest`
- 会生成报告

但一旦别人问：

- 你的项目结构是什么？
- 你测了哪些接口？
- 你怎么组织配置和数据？
- 你怎么运行这套测试？
- 你这个项目的价值是什么？

就讲不清楚。

Day 7 的作用就是补这个缺口。

它让你从：

- “我学过这些知识点”

变成：

- “我已经有一套能展示、能讲解、能继续扩展的小项目”

## 7. Day 7 应该怎么划定范围

一个常见错误是 Day 7 还在继续无限加功能。

这会导致：

- 今天做不完
- 项目越来越散
- 最终没有一个完整收尾

所以 Day 7 更推荐先定一个最小范围。

### 7.1 推荐测试范围

建议至少覆盖这些接口：

1. `GET /health`
2. `POST /api/v1/login`
3. `GET /api/v1/profile`
4. `GET /api/v1/products`
5. `GET /api/v1/products/{id}`
6. `POST /api/v1/products`
7. `DELETE /api/v1/products/{id}`

### 7.2 推荐测试场景

建议至少覆盖：

#### 正向场景

- 健康检查成功
- 登录成功
- 查询个人资料成功
- 查询商品列表成功
- 查询存在商品成功
- 创建商品成功
- 删除商品成功

#### 反向场景

- 登录失败
- 查询不存在商品
- 删除后再次查询商品返回不存在
- 用户名为空
- 用户不存在

### 7.3 为什么这样定范围

因为这个范围刚好覆盖了你前几天学到的核心能力：

- 基础接口请求
- 参数化测试
- fixture
- token 关联
- 正反向断言
- 业务链路测试

这不是“为了凑数量”，而是为了让 Day 7 的成品既完整，又不过大。

## 8. Day 7 的项目结构应该怎么看

你现在的 Day 7 成品，建议至少有下面这些内容：

```text
api_auto_test/
├── common/
│   ├── request_handler.py
│   ├── logger.py
│   └── assert_util.py
├── config/
│   ├── settings.py
│   └── env.json
├── data/
│   ├── login_cases.json
│   └── new_product_case.json
├── testcases/
│   ├── test_health.py
│   ├── test_login.py
│   ├── test_profile.py
│   ├── test_product.py
│   ├── test_product_list.py
│   └── test_product_create_query_delete.py
├── reports/
├── conftest.py
├── pytest.ini
└── README.md
```

### 8.1 不要只会背目录名

Day 7 你不能只说：

- 这里放配置
- 那里放测试

你还要能讲清楚这样组织后的效果：

- `config/`：统一管理环境和账号配置，改动范围小
- `common/`：统一请求、日志、断言入口，减少重复代码
- `data/`：测试数据和测试逻辑分离，便于参数化和维护
- `testcases/`：测试文件专注业务意图
- `conftest.py`：统一管理前置步骤和公共 fixture
- `pytest.ini`：统一测试发现规则和运行配置
- `reports/`：保存测试结果产物
- `README.md`：告诉别人怎么运行、怎么理解这套项目

也就是说，Day 7 不只是“会放”，而是“会解释为什么这样放”。

## 9. README 为什么必须有

这是 Day 7 最容易被忽略，但实际非常重要的一块。

如果没有 README，别人拿到你的项目时会立刻遇到问题：

- 这是什么项目？
- 怎么启动本地服务？
- 怎么运行测试？
- 用了什么技术栈？
- 已经实现了哪些能力？

所以 README 不是装饰品，而是项目说明书。

### 9.1 最小 README 应该包含什么

建议至少包含：

1. 项目简介
2. 技术栈
3. 项目结构
4. 运行前准备
5. 本地服务启动方式
6. 测试执行方式
7. 报告生成方式
8. 已覆盖接口和场景

### 9.2 一个最小 README 结构示例

```md
# 项目名称

## 1. 项目简介

## 2. 技术栈

## 3. 项目结构

## 4. 环境准备

## 5. 启动本地接口服务

## 6. 执行测试

## 7. 生成报告

## 8. 已实现能力
```

### 9.3 README 的实际价值

它会直接影响别人对你项目的第一印象：

- 没有 README：像练习碎片
- 有 README：像一个可交付的小项目

## 10. Day 7 的项目不只是“能跑”，还要“能讲”

这是今天最重要的目标之一。

很多人能跑测试，但一讲项目就很空。

所以 Day 7 你要能把下面这些问题讲清楚：

1. 这个项目测的是什么业务
2. 为什么选这些接口
3. 怎么组织配置、数据、公共方法
4. 登录 token 是怎么关联的
5. 正向和反向场景怎么覆盖的
6. 报告和日志怎么帮助定位问题

### 10.1 你至少应该能讲出的项目介绍

比如你可以这样介绍：

“这是一个基于 Python、requests、pytest 的接口自动化练习项目。  
我基于本地 Flask 服务实现了登录、用户资料、商品查询、商品新增删除等接口的自动化测试。  
项目里把配置、测试数据、公共请求入口、日志和断言做了基础分层，支持参数化测试、token 鉴权、链路测试和 HTML 报告生成。”  

这段话不需要死背，但你最好能讲到这个程度。

## 11. Day 7 最推荐保留的测试能力

Day 7 不求花哨，但至少应保留这些“能体现你已经掌握接口自动化”的能力。

### 11.1 参数化

例如登录测试：

- 成功登录
- 密码错误
- 用户名为空
- 用户不存在

这能体现你不是只会写单条用例。

### 11.2 fixture

例如：

- `reset_data`
- `login_token`
- `auth_headers`

这能体现你已经开始把前置动作和测试逻辑分离。

### 11.3 接口关联

例如：

1. 登录拿 token
2. 带 token 查资料
3. 创建商品拿 `id`
4. 用 `id` 查询和删除商品

这能体现你已经会写链路测试，不只是独立接口测试。

### 11.4 报告和日志

这能体现你已经考虑失败定位，而不是只会看终端输出。

## 12. Day 7 的交付标准应该怎么理解

今天最容易走偏的地方是：

- 只顾着多写几个测试
- 忘了项目要能展示

所以 Day 7 的交付标准建议分成 3 层。

### 12.1 第一层：能运行

至少要满足：

- 能启动本地服务
- 能执行 `pytest`
- 能生成测试报告

### 12.2 第二层：能维护

至少要满足：

- 配置不散落
- 测试数据和测试逻辑分离
- 公共请求入口统一
- fixture 可复用

### 12.3 第三层：能说明

至少要满足：

- 有 README
- 能讲清楚项目结构
- 能讲清楚覆盖范围
- 能讲清楚你做了哪些设计取舍

Day 7 最终更接近第三层，而不是只停在第一层。
