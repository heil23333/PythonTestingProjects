## 5. 讲义模块二：检查项目结构

你今天应该先对照下面这份结构，看看自己缺什么：

```text
api_auto_test/
├── common/
├── config/
├── data/
├── testcases/
├── reports/
├── conftest.py
├── pytest.ini
└── README.md
```

### 5.1 今天最少要检查什么

1. `config/` 是否已有配置文件
2. `common/` 是否已有请求、日志、断言
3. `data/` 是否已有登录和商品数据
4. `testcases/` 是否已有主要测试文件
5. `README.md` 是否存在

### 5.2 你要理解

今天检查结构不是为了“看着整齐”，而是为了确认：

这个项目是不是已经具备可交付的骨架。

### 5.3 结构检查后的动作

如果你发现：

- 代码能跑，但没有 README

那今天优先补 README。

如果你发现：

- 测试都在，但报告目录没有

那今天优先补报告产物。

也就是说：

Day 7 更像查漏补缺。

## 6. 讲义模块三：梳理测试范围

今天建议你把已有测试分成两类：

### 6.1 基础接口测试

例如：

- `test_health.py`
- `test_login.py`
- `test_profile.py`
- `test_product.py`
- `test_product_list.py`

### 6.2 链路测试

例如：

- `test_product_create_query_delete.py`

### 6.3 你要理解

这样分的作用是：

- 基础接口测试体现覆盖面
- 链路测试体现业务流程能力

两类都保留，项目会更完整。

## 7. 讲义模块四：补 README

这是今天最重要的动作之一。

### 7.1 README 至少应该包含

1. 项目简介
2. 技术栈
3. 项目结构
4. 启动本地服务方式
5. 测试执行方式
6. 报告生成方式
7. 已覆盖接口和场景

### 7.2 一个最小 README 模板

```md
# 用户与商品接口自动化测试项目

## 1. 项目简介

## 2. 技术栈

## 3. 项目结构

## 4. 启动本地接口服务

## 5. 执行测试

## 6. 生成测试报告

## 7. 已覆盖接口
```

### 7.3 你要理解

README 的价值不是“凑文档”，而是：

让别人拿到你的项目后知道怎么跑、怎么理解。

### 随堂检查

1. 为什么 Day 7 没有 README 会很吃亏？

### 参考答案

1. 因为项目即使能跑，没有 README 别人也不知道这是什么、怎么启动、怎么执行、覆盖了什么。

## 8. 讲义模块五：跑一次完整测试

今天要把整套测试作为成品跑一遍。

### 8.1 最小命令

```bash
pytest
```

### 8.2 生成报告

```bash
mkdir -p reports
pytest --html=reports/report.html --self-contained-html
```

### 8.3 你要理解

Day 7 的报告不是“随手生成一下”，而是最终交付物的一部分。

也就是说：

- 你不只是验证测试能跑
- 你还在产出项目结果

## 9. 讲义模块六：整理一段项目介绍

今天非常建议你给自己准备一段 30 秒到 1 分钟的项目介绍。

### 9.1 推荐结构

4. 做了哪些分层设计

### 9.2 一个可参考版本

```text
这是一个基于 Python、requests、pytest 的接口自动化练习项目。
我基于本地 Flask 服务，对登录、用户资料、商品查询和商品新增删除等接口做了自动化测试。
项目里把配置、测试数据、公共请求、日志、断言和测试用例做了基础分层，支持参数化、fixture、token 鉴权和链路测试，并且可以生成 HTML 报告。
```

### 9.3 你要理解

Day 7 的目标不只是“你心里知道自己做了什么”，而是“你能把它清楚说出来”。
