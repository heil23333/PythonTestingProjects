## 5. 什么叫配置分离

配置分离，就是把经常变化但又不属于测试逻辑本身的内容，单独放出去。

最常见的配置有：

- `BASE_URL`
- 默认账号
- 默认密码
- 超时时间
- 测试环境标识

### 5.1 为什么要分离

如果你把这些都写死在测试文件里，会有几个问题：

- 改地址时要改很多文件
- 账号变了要到处搜
- 测试代码和环境信息混在一起

### 5.2 Day 4 推荐的最小方案

你现在不用一上来就上 `yaml`。

最稳的方式是先用一个简单的 `config.py`。

例如：

```python
BASE_URL = "http://127.0.0.1:5001"
USERNAME = "alice"
PASSWORD = "123456"
TIMEOUT = 5
```

然后测试里直接导入：

```python
from config import BASE_URL
```

### 5.3 为什么这里不用 fixture

`BASE_URL` 本质上是固定配置，不是动态数据。

所以：

- 常量配置适合放 `config.py`
- 动态前置数据适合放 fixture

这个区分很重要。

## 6. 什么叫测试数据分离

测试数据分离，就是把“我要测哪些输入和预期”从测试逻辑里拆出去。

例如登录测试里，下面这些就是测试数据：

- 用户名
- 密码
- 预期状态码
- 预期业务码
- 预期消息

### 6.1 为什么要分离

如果你把大量测试数据直接塞在测试函数里：

- 文件会变长
- 数据和逻辑混在一起
- 维护不方便

### 6.2 Day 4 推荐的最小方案

你现在有 2 种常见方案：

#### 方案 1：放到 Python 文件里

例如：

```python
LOGIN_CASES = [
    {
        "username": "alice",
        "password": "123456",
        "expected_status": 200,
        "expected_code": 0,
        "expected_message": "success",
    },
    {
        "username": "alice",
        "password": "wrong",
        "expected_status": 401,
        "expected_code": 10014,
        "expected_message": "password error",
    },
]
```

优点：

- 最简单
- 不用额外解析文件

#### 方案 2：放到 JSON 文件里

例如：

```json
[
  {
    "username": "alice",
    "password": "123456",
    "expected_status": 200,
    "expected_code": 0,
    "expected_message": "success"
  },
  {
    "username": "alice",
    "password": "wrong",
    "expected_status": 401,
    "expected_code": 10014,
    "expected_message": "password error"
  }
]
```

优点：

- 更像真实项目
- 数据和代码分得更开

### 6.3 你当前最推荐怎么做

Day 4 我建议你直接练 JSON 文件。

因为后面做接口自动化时，测试数据独立文件是常见模式。

## 7. 什么叫接口关联

接口关联，就是前一个接口的返回结果，被后一个接口继续使用。

这在真实项目里非常常见。

### 7.1 Day 4 最典型的两个关联

#### 关联一：登录拿 token

流程：

1. 调登录接口
2. 从响应中提取 `token`
3. 把 token 放到后续请求头里

#### 关联二：新增商品拿 id

流程：

1. 调新增商品接口
2. 从响应中提取 `id`
3. 用这个 `id` 去查询详情
4. 再用这个 `id` 去删除

### 7.2 为什么这一步重要

如果不会接口关联，你的自动化测试就只能停在“单接口”层面。

而真实业务最有价值的往往是链路测试。

## 8. 什么叫断言设计

断言不是“只写一个 `status_code == 200`”。

好的断言应该有层次。

### 8.1 推荐的 4 层断言

#### 第一层：HTTP 状态码

```python
assert resp.status_code == 200
```

#### 第二层：业务码

```python
assert body["code"] == 0
```

#### 第三层：业务消息

```python
assert body["message"] == "success"
```

#### 第四层：关键业务字段

```python
assert body["data"]["username"] == "alice"
assert body["data"]["id"] == product_id
```

### 8.2 为什么断言要有层次

因为这能帮你快速定位问题属于哪一层：

- HTTP 层
- 业务层
- 数据层

### 10.3 `conftest.py`

```python
import pytest
import requests

from config import BASE_URL, PASSWORD, TIMEOUT, USERNAME


@pytest.fixture
def reset_data():
    requests.post(f"{BASE_URL}/api/v1/debug/reset", timeout=TIMEOUT)


@pytest.fixture
def login_token(reset_data):
    resp = requests.post(
        f"{BASE_URL}/api/v1/login",
        json={"username": USERNAME, "password": PASSWORD},
        timeout=TIMEOUT,
    )
    body = resp.json()

    assert resp.status_code == 200
    assert body["code"] == 0

    return body["data"]["token"]


@pytest.fixture
def auth_headers(login_token):
    return {"Authorization": f"Bearer {login_token}"}
```

### 10.4 `test_login.py`

```python
import json
from pathlib import Path

import pytest
import requests

from config import BASE_URL, TIMEOUT


DATA_FILE = Path(__file__).resolve().parent / "data" / "login_cases.json"
LOGIN_CASES = json.loads(DATA_FILE.read_text(encoding="utf-8"))


@pytest.mark.parametrize("case", LOGIN_CASES)
def test_login(case):
    resp = requests.post(
        f"{BASE_URL}/api/v1/login",
        json={
            "username": case["username"],
            "password": case["password"],
        },
        timeout=TIMEOUT,
    )
    body = resp.json()

    assert resp.status_code == case["expected_status"]
    assert body["code"] == case["expected_code"]
    assert body["message"] == case["expected_message"]

    if case["expected_code"] == 0:
        assert "token" in body["data"]
    else:
        assert body["data"] is None
```

### 10.5 `test_product_flow.py`

```python
import requests

from config import BASE_URL, TIMEOUT


def test_product_create_query_delete(auth_headers):
    create_resp = requests.post(
        f"{BASE_URL}/api/v1/products",
        headers=auth_headers,
        json={
            "name": "Desk Lamp",
            "category": "lighting",
            "price": 199,
        },
        timeout=TIMEOUT,
    )
    create_body = create_resp.json()

    assert create_resp.status_code == 201
    assert create_body["code"] == 0
    assert create_body["message"] == "success"

    product_id = create_body["data"]["id"]

    detail_resp = requests.get(
        f"{BASE_URL}/api/v1/products/{product_id}",
        timeout=TIMEOUT,
    )
    detail_body = detail_resp.json()

    assert detail_resp.status_code == 200
    assert detail_body["code"] == 0
    assert detail_body["data"]["id"] == product_id
    assert detail_body["data"]["name"] == "Desk Lamp"

    delete_resp = requests.delete(
        f"{BASE_URL}/api/v1/products/{product_id}",
        headers=auth_headers,
        timeout=TIMEOUT,
    )
    delete_body = delete_resp.json()

    assert delete_resp.status_code == 200
    assert delete_body["code"] == 0
    assert delete_body["data"]["deleted"] is True
```

## 12. `BASE_URL`、账号密码、token 应该分别放哪

这个区分很重要。

### 12.1 `BASE_URL`

属于固定配置，放 `config.py`

### 12.2 账号密码

当前阶段可以放 `config.py`

如果后面有很多账号组合，也可以放测试数据文件

### 12.3 token

属于动态运行时数据，不放配置文件。

应该通过：

- fixture
- 前置请求
- 响应提取

在运行时生成。

---

## 13. 为什么 `reset_data` 很重要

你本地接口服务里专门有：

```text
POST /api/v1/debug/reset
```

它的价值很大，因为它可以让测试回到固定初始状态。

这意味着：

- 你新增过的数据可以清掉
- 旧 token 可以失效
- 每次测试环境更稳定

这就是“测试前置数据治理”的最小雏形。

## 14. Day 4 常见错误

### 14.1 把所有东西都塞进 fixture

这会让逻辑越来越难懂。

要记住：

- 固定配置 -> `config.py`
- 动态前置 -> fixture
- 输入输出数据 -> data 文件

### 14.2 参数化数据还全写在测试函数里

Day 4 就要开始有意识拆出去。

### 14.3 只断言状态码

这不够。

至少还要断业务码和关键字段。

### 14.4 链路测试不做清理

如果新增后不删除，或者不 reset，数据会越来越脏。

### 14.5 每个测试都重复登录

重复登录不是不行，但会让代码非常啰嗦。

应该逐步把它收敛到 fixture。
