# Python 接口自动化测试 7 天速成计划

## 1. 适用人群

这份计划适合你当前这种情况：

- 已有软件开发基础
- 熟悉基础编程概念、模块化、调试、命令行
- 希望快速切入 Python 接口自动化测试
- 目标是 7 天内搭出一个能跑的接口自动化小项目

这不是“零基础入门版”，而是“快速迁移版”。

---

## 2. 7 天后的目标

完成 7 天学习后，你至少要达到下面这个水平：

- 能看懂接口文档并拆出测试点
- 能用 `requests` 编写接口请求
- 能用 `pytest` 编写参数化测试和公共 fixture
- 能处理登录鉴权、接口关联、环境配置
- 能搭一个基础接口自动化项目骨架
- 能输出测试报告并执行一套真实业务接口用例

---

## 3. 学习原则

- 每天以“学完即写”为准，不做纯阅读
- 每天至少产出代码，不只记笔记
- 每天结束都要有可运行结果
- 7 天目标不是做“大而全框架”，而是做“小而完整项目”

建议每天投入时间：

- 最低 2 小时
- 理想 3 到 5 小时

---

## 4. 7 天详细安排

## Day 1：补齐接口测试最小知识集

### 当天目标

建立接口测试的核心认知，并能手工调接口。

### 学习内容

- HTTP 基础快速过一遍
  - `GET`、`POST`、`PUT`、`DELETE`
  - `headers`、`params`、`json`、`data`
  - 状态码
  - Cookie、Session、Token、JWT
- 接口测试测什么
  - 功能正确性
  - 参数校验
  - 业务规则
  - 鉴权控制
  - 异常场景
- 熟悉一个接口调试工具
  - Postman 或 Apifox

### 当天任务

1. 找一套公开接口或已有练习接口
   - 推荐优先使用：
     - `Postman Echo`
     - `httpbin`
     - `JSONPlaceholder`
     - `DummyJSON`
2. 手工调通以下接口类型
   - 登录
   - 查询列表
   - 查询详情
   - 新增
   - 删除
3. 记录每个接口的
   - URL
   - 请求方法
   - 请求头
   - 请求参数
   - 成功响应
   - 失败响应

### 当天输出

- 一份接口清单
- 一份测试点草稿
- 一个 Postman/Apifox 集合

### 推荐接口使用顺序

1. 先用 `Postman Echo` 和 `httpbin` 练 HTTP 基础
2. 再用 `JSONPlaceholder` 练 CRUD
3. 最后用 `DummyJSON` 练登录和鉴权

### 如果没有可用接口

如果公开接口不可用，或者不适合当前学习目标，就不应该停在理论阶段。

正确做法是：

1. 直接创建一套本地可运行的练习接口代码
2. 再基于本地接口继续做手工测试和自动化测试

后续学习默认按这个原则处理。

### 验收标准

- 你能不看教程，自己解释清楚一个接口怎么调用
- 你能说出至少 5 个接口测试点

---

## Day 2：用 Python + requests 接管接口调用

### 当天目标

把手工调接口变成 Python 代码。

### 学习内容

- 安装并使用 `requests`
- `requests.get/post/put/delete`
- 响应处理
  - `status_code`
  - `text`
  - `json()`
- 请求参数传递
  - `params`
  - `data`
  - `json`
- 公共请求头
- `Session` 会话保持
- 超时和异常处理

### 当天任务

1. 用 Python 改写 Day 1 里至少 4 个接口
2. 写一个公共请求函数，例如：
   - `send_request(method, url, **kwargs)`
3. 实现登录并保存 token
4. 用 token 调用一个需要鉴权的接口

### 建议代码目标

- 单文件先跑通
- 先不要过度设计
- 只做最小封装

### 当天输出

- 一个 `requests` 练习脚本
- 一个通用请求方法
- 一个登录后访问业务接口的示例

### 验收标准

- 你能不用 Postman，直接靠 Python 调通接口
- 你能解释 `params/data/json` 的区别

---

## Day 3：切到 pytest，建立测试用例结构

### 当天目标

把“接口脚本”升级成“可执行测试”。

### 学习内容

- `pytest` 基本规则
- `assert`
- 测试文件与测试函数命名
- fixture
- 参数化 `@pytest.mark.parametrize`
- `conftest.py`
- `pytest.ini`

### 当天任务

1. 安装 `pytest`
2. 把 Day 2 的接口脚本改造成测试用例
3. 为登录逻辑封装一个 fixture
4. 为同一个接口设计 3 到 5 组参数化数据
5. 运行测试并观察断言输出

### 推荐最小目录

```text
project/
├── common/
├── data/
├── testcases/
├── conftest.py
└── pytest.ini
```

### 当天输出

- 至少 5 个 `pytest` 接口测试用例
- 1 个登录 fixture
- 1 个参数化测试文件

### 验收标准

- 你能解释 fixture 的作用
- 你能把重复测试逻辑抽出来

---

## Day 4：补上接口自动化真正关键的能力

### 当天目标

掌握真实项目最常见的 4 个问题：断言、关联、数据、配置。

### 学习内容

- 断言设计
  - 状态码断言
  - 响应字段断言
  - 业务断言
- 接口关联
  - 提取 `token`
  - 提取 `id`
  - 前一个接口输出作为后一个接口输入
- 测试数据管理
  - `json` 或 `yaml`
- 环境配置管理
  - `base_url`
  - 测试账号

### 当天任务

1. 把测试数据从代码里拆出去
2. 把 `base_url`、账号密码放到配置文件
3. 做一个完整链路：
   - 登录
   - 创建数据
   - 查询数据
   - 删除数据
4. 每一步都加明确断言

### 当天输出

- 配置文件
- 测试数据文件
- 一条完整业务链路用例

### 验收标准

- 你能处理“先创建再查询再删除”的依赖场景
- 你能区分“请求成功”和“业务正确”

## Day 5：搭基础框架骨架

### 当天目标

把散落代码整理成一个可维护的接口自动化小项目。

### 学习内容

- 常见项目分层
- 请求封装
- 配置读取
- 日志输出
- 断言封装
- 数据读取

### 推荐目录结构

```text
api_auto_test/
├── common/
│   ├── request_handler.py
│   ├── logger.py
│   ├── config_util.py
│   └── assert_util.py
├── config/
│   └── config.yaml
├── data/
│   └── login_data.yaml
├── testcases/
│   ├── test_login.py
│   └── test_business_flow.py
├── conftest.py
├── pytest.ini
└── requirements.txt
```

### 当天任务

1. 重构 Day 1 到 Day 4 的代码
2. 把请求逻辑收敛到公共模块
3. 加日志打印
4. 加简单断言工具方法
5. 确保执行入口清晰

### 当天输出

- 一个结构完整的小型接口自动化项目
- 至少 2 个业务测试文件

### 验收标准

- 目录结构不是“所有代码都堆一起”
- 新增一个接口用例时，改动成本较低

## Day 6：报告、失败定位、项目增强

### 当天目标

让测试结果可读、可定位，而不是“跑完只知道失败了”。

### 学习内容

- `pytest-html` 或 `allure` 基础
- 日志定位思路
- 请求和响应打印
- 失败时保留上下文
- 标记用例
  - `smoke`
  - `regression`

### 当天任务

1. 集成一种报告方案
2. 为关键步骤添加日志
3. 区分冒烟与回归用例
4. 故意制造一个失败用例，检查是否容易定位问题

### 当天输出

- 一份测试报告
- 一套失败可定位的日志
- 冒烟用例标签

### 验收标准

- 失败后你能快速看到请求参数、响应结果和断言点
- 你能只执行某一类用例

## 9. 你的最佳执行策略

基于你“有软件开发基础”这个前提，最有效的方式不是补太多 Python 语法，而是：

1. 用 1 天把接口测试认知补齐
2. 用 2 天把 `requests + pytest` 跑熟
3. 用 2 天做关联、配置、数据分离
4. 用 2 天完成小项目和报告

你的重点应该是：

- 测试思维
- 接口测试点设计
- `pytest` 组织方式
- 自动化项目结构

而不是卡在语法细节。
