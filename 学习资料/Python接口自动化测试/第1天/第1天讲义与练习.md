## 4. 讲义模块一：接口到底是什么

### 4.1 核心定义

接口可以理解成系统对外暴露的功能入口。

你可以把它看成系统的一扇标准化窗口。外部系统、前端页面、移动端或者测试脚本，都通过这个窗口与系统交互。

常见接口例子：

- 登录接口
- 查询用户信息接口
- 创建订单接口
- 删除购物车商品接口

### 4.2 接口为什么是测试重点

因为很多真实业务能力都在接口层完成：

- 前端按钮点击后，最终调的是接口
- App 页面加载数据，最终调的是接口
- 系统与系统之间联动，也常常调接口

接口层稳定，说明系统主干能力通常比较稳定。

### 4.3 你现在最需要建立的理解

第 1 天先不要把接口想得太复杂。

你只需要先把接口理解成：

- 你发一个请求
- 系统处理这个请求
- 系统返回一个结果
- 你根据结果判断接口是否正确

### 4.4 随堂检查

1. 接口和页面的核心区别是什么？
2. 为什么前端自动化不能替代接口测试？
3. 举 3 个你熟悉的业务接口例子。

### 4.5 标准答案

1. 页面关注展示和交互，接口关注数据和逻辑。
2. 因为 UI 层只能看到结果表现，接口层更靠近真实业务逻辑，执行更快、定位更直接。
3. 例如登录、查询订单、提交评论。

## 5. 讲义模块二：什么是接口测试

### 5.1 定义

接口测试的本质，是验证系统接口是否按照预期工作。

它不是只验证“能不能调通”，而是验证：

- 请求是否正确
- 参数是否被正确处理
- 返回结果是否正确
- 业务逻辑是否正确
- 权限控制是否正确
- 异常输入是否被妥善拦截

### 5.2 接口测试主要测什么

常见检查点包括：

- 功能正确性
- 参数合法性
- 业务规则
- 鉴权控制
- 异常场景
- 数据一致性

### 5.3 接口测试和 UI 测试的区别

| 对比项 | 接口测试 | UI 测试 |
| --- | --- | --- |
| 主要验证对象 | 接口逻辑和数据 | 页面交互和展示 |
| 执行速度 | 快 | 慢 |
| 稳定性 | 高 | 相对低 |
| 定位问题 | 更直接 | 更间接 |
| 维护成本 | 较低 | 较高 |

### 5.4 初学者最容易犯的错误

- 只看状态码，不看响应业务字段
- 只测成功场景，不测失败场景
- 只知道调接口，不知道为什么这样测

### 5.5 随堂检查

判断对错：

1. 只要返回 `200`，接口就算测试通过。
2. 接口测试只需要关注是否有返回值。
3. 登录失败场景也属于接口测试范围。

### 5.6 标准答案

1. 错。`200` 只说明 HTTP 层成功，不一定代表业务成功。
2. 错。还要看返回内容是否符合预期。
3. 对。异常场景是接口测试的重要组成部分。

## 6. 讲义模块三：HTTP 基础

### 6.1 HTTP 的本质

HTTP 是客户端和服务器通信的协议。

在接口测试里，你可以把它理解成一套固定的交流规则：

- 客户端发请求
- 服务器收请求
- 服务器返回响应

### 6.2 一次接口调用的过程

1. 你准备 URL、请求方法、请求参数、请求头
2. 你发起请求
3. 服务端收到请求并处理
4. 服务端返回状态码、响应头、响应体
5. 你分析结果

### 6.3 HTTP 和 HTTPS

- HTTP：明文传输
- HTTPS：加密传输

真实项目里大多数接口都使用 HTTPS。

### 6.4 URL 的组成

示例：

```text
https://api.example.com/users/1001?page=1
```

可拆成：

- 协议：`https`
- 域名：`api.example.com`
- 路径：`/users/1001`
- 查询参数：`page=1`

### 6.5 随堂检查

1. URL 中哪一部分通常表示“要访问哪个资源”？
2. `https://` 和 `http://` 的主要区别是什么？
3. 查询参数一般出现在哪一部分？

### 6.6 标准答案

1. 路径部分。
2. HTTPS 在传输层面更安全，带加密。
3. 一般出现在 URL 末尾的查询串里。

## 7. 讲义模块四：HTTP 请求方法

### 7.1 GET

常见用途：

- 查询列表
- 查询详情

典型特点：

- 一般不修改数据
- 参数常放在 URL 上

### 7.2 POST

常见用途：

- 登录
- 注册
- 提交表单
- 新增数据

典型特点：

- 参数常放在请求体里

### 7.3 PUT / PATCH

常见用途：

- 修改数据

一般理解：

- `PUT` 更偏整体更新
- `PATCH` 更偏局部更新

### 7.4 DELETE

常见用途：

- 删除数据

### 7.5 现实项目要注意什么

虽然 RESTful 风格推荐不同方法承担不同语义，但真实项目里并不总规范。

比如：

- 有的系统用 `POST` 做查询
- 有的系统用 `POST` 做删除

因此你的判断依据应该是：

1. 接口文档
2. 真实服务实现
3. 团队约定

### 7.6 随堂检查

为下面场景匹配更常见的方法：

1. 查询订单详情
2. 创建新用户
3. 修改用户手机号
4. 删除一条评论

### 7.7 标准答案

1. `GET`
2. `POST`
3. `PUT` 或 `PATCH`
4. `DELETE`

## 8. 讲义模块五：请求由哪些部分组成

### 8.1 请求的核心要素

一条接口请求至少要看清这几个要素：

- URL
- Method
- Headers
- Params
- Body

### 8.2 Headers

Headers 是请求头，用来描述附加信息。

常见请求头：

- `Content-Type`
- `Authorization`
- `Cookie`

示例：

```http
Content-Type: application/json
Authorization: Bearer token123
```

### 8.3 Params

`params` 一般是 URL 上的查询参数。

例子：

```text
GET /products?page=1&page_size=10
```

这里的 `page` 和 `page_size` 就是查询参数。

### 8.4 Body

Body 就是请求体，常用于提交数据。

典型场景：

- 登录
- 注册
- 创建订单
- 修改资料

### 8.5 data 和 json 的区别

#### `data`

通常是表单格式。

比如：

```text
username=tom&password=123456
```

#### `json`

通常是 JSON 格式。

比如：

```json
{
  "username": "tom",
  "password": "123456"
}
```

#### 先记场景，不急着记底层细节

- 查询类接口：更常见 `params`
- 提交类接口：更常见 `json` 或 `data`

### 8.6 一个最常见误区

很多人一开始分不清 `params` 和 `json`。

你先只记这一句：

- `params` 多数拼在 URL 上
- `json` 多数放在请求体里

后面你用 `requests` 时会彻底吃透。

### 8.7 随堂检查

判断下面内容更可能属于哪一类：

1. `?page=1&page_size=20`
2. `Authorization: Bearer abc`
3. `{"username": "tom", "password": "123456"}`

### 8.8 标准答案

1. `params`
2. `headers`
3. `json`

## 9. 讲义模块六：响应由哪些部分组成

### 9.1 响应的核心内容

接口响应通常包含：

- 状态码
- 响应头
- 响应体

### 9.2 状态码

状态码表示 HTTP 协议层面的结果。

常见状态码：

- `200`：成功
- `201`：创建成功
- `400`：请求错误
- `401`：未认证
- `403`：无权限
- `404`：资源不存在
- `500`：服务端异常

### 9.3 响应体

响应体通常是 JSON。

例如：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "user_id": 1001
  }
}
```

### 9.4 状态码和业务码不是一回事

这个认知非常关键。

例如登录失败接口也可能返回：

```json
{
  "code": 10002,
  "message": "password error",
  "data": null
}
```

但 HTTP 状态码仍然可能是 `200`。

因此测试时要至少看两层：

1. 协议层是否成功
2. 业务层是否成功

### 9.5 随堂检查

1. `401` 通常说明什么？
2. `500` 更可能是客户端问题还是服务端问题？
3. 一个接口返回 `200`，但业务码是失败码，这个接口能不能算业务成功？

### 9.6 标准答案

1. 通常说明未认证，例如未登录、token 无效、token 过期。
2. 更可能是服务端问题。
3. 不能算业务成功。

## 10. 讲义模块七：Cookie、Session、Token、JWT

### 10.1 为什么会有认证机制

很多接口不能随便访问，比如：

- 查询个人订单
- 删除数据
- 管理后台接口

系统必须知道“当前是谁在请求”，所以需要认证机制。

### 10.2 Cookie

Cookie 是服务端发给客户端的一小段标识信息，客户端后续访问时会带回去。

### 10.3 Session

Session 是服务端保存的会话状态。

通常：

1. 用户登录
2. 服务端创建 Session
3. 客户端保存一个标识
4. 后续请求带着标识访问

### 10.4 Token

Token 是现代接口常用的认证方式。

典型流程：

1. 登录成功
2. 服务端返回 token
3. 后续请求把 token 放到请求头里

示例：

```http
Authorization: Bearer xxxxx
```

### 10.5 JWT

JWT 是一种 Token 形式。

你现在不用深究它的编码结构，只要先知道：

- 它也是身份凭证
- 很多系统用它做认证
- 它通常有过期时间

### 10.6 认证类接口要怎么测

至少要覆盖：

- 登录后访问成功
- 未登录访问失败
- token 错误访问失败
- token 过期访问失败
- 普通用户越权访问管理员接口失败

### 10.7 随堂检查

1. 登录成功后最常拿到什么用于后续请求？
2. `Authorization` 请求头通常和什么概念关联？
3. 为什么“未登录访问受限接口”必须测？

### 10.8 标准答案

1. Token。
2. 身份认证。
3. 因为权限控制是接口安全的重要部分。

## 11. 讲义模块八：如何看接口文档

### 11.1 一份接口文档至少看什么

你阅读接口文档时，重点看这些字段：

- 接口名称
- 请求地址
- 请求方法
- 是否需要鉴权
- 请求头
- 请求参数
- 请求示例
- 成功响应
- 失败响应
- 错误码说明

### 11.2 推荐阅读顺序

每次都按固定顺序读，效率会高很多：

1. 这个接口干什么
2. 用什么方法请求
3. 地址是什么
4. 参数放在哪里
5. 需不需要登录
6. 成功时返回什么
7. 失败时返回什么
8. 我应该测哪些点

### 11.3 文档不准怎么办

真实项目里文档经常不够准确。

你应该这样处理：

1. 先按文档调
2. 再按真实返回验证
3. 如果不一致，记录并和开发确认

### 11.4 随堂检查

看到一个接口文档时，你最先要看的是哪 3 个信息？

### 11.5 参考答案

通常先看：

1. 请求方法
2. 请求地址
3. 请求参数和是否需要鉴权

## 12. 讲义模块九：如何手工调用一个接口

### 12.1 工具选择

今天只需要掌握一种：

- Postman
- Apifox

### 12.2 标准操作步骤

1. 选择请求方法
2. 输入 URL
3. 填写 Headers
4. 填写 Params 或 Body
5. 点击发送
6. 查看状态码
7. 查看响应体
8. 判断是否符合预期

### 12.3 以登录接口为例

假设接口定义如下：

- URL：`https://api.example.com/api/v1/login`
- Method：`POST`
- Header：`Content-Type: application/json`
- Body：

```json
{
  "username": "tom",
  "password": "123456"
}
```

你应完成：

1. 设置 `POST`
2. 输入 URL
3. Headers 配 `Content-Type`
4. Body 选择 JSON
5. 输入请求体
6. 发送并查看是否返回 token

### 12.4 以查询详情接口为例

- URL：`https://api.example.com/api/v1/users/1001`
- Method：`GET`
- Header：`Authorization: Bearer token123`

你应完成：

1. 设置 `GET`
2. 输入 URL
3. 填 token
4. 点击发送
5. 检查用户信息是否正确

### 12.5 随堂检查

1. 手工调接口时，为什么只看状态码不够？
2. 查询详情接口通常更常用什么请求方法？
3. 登录接口更常见把参数放在哪里？

### 12.6 标准答案

1. 因为业务字段也可能失败。
2. `GET`
3. 请求体里，常见为 `json` 或 `data`。

## 13. 讲义模块十：接口测试点怎么拆

### 13.1 五大类测试点

你至少按下面 5 类来拆：

- 正常功能
- 参数校验
- 业务规则
- 鉴权控制
- 异常场景

### 13.2 登录接口示例

接口：用户登录

可拆测试点：

- 正确用户名 + 正确密码
- 用户名为空
- 密码为空
- 用户名不存在
- 密码错误
- 字段缺失
- 请求方法错误
- Content-Type 错误

### 13.3 查询接口示例

接口：查询商品列表

可拆测试点：

- 正常分页查询
- 页码为 0
- 页码为负数
- 每页条数过大
- 参数类型错误
- 无数据查询

### 13.4 新增接口示例

- 必填字段全部正确
- 缺少必填字段
- 字段格式错误
- 重复创建
- 创建成功后再查询验证

### 13.5 删除接口示例

- 删除存在数据
- 删除不存在数据
- 未登录删除
- 无权限删除
- 删除后再查是否不存在

### 13.6 一个非常重要的认知

测试点不是越多越好，而是要有结构。

一个好的初始测试点集合，应该保证：

- 有正常场景
- 有典型异常场景
- 有权限场景
- 有业务规则场景

### 13.7 随堂检查

给“查询订单详情接口”至少列 5 个测试点。

### 13.8 参考示例

- 查询存在订单成功

## 15. 配套练习一：概念题

请先独立作答，再看答案。

### 题目

1. 什么是接口？
2. 接口测试和 UI 测试最大的差异是什么？
3. `GET` 和 `POST` 各自更常见的用途是什么？
4. `params` 和 `json` 的常见区别是什么？
5. 为什么状态码正常不等于业务成功？

### 参考答案

1. 接口是系统对外暴露的功能调用入口。
2. 接口测试更关注数据和逻辑，UI 测试更关注页面和交互。
3. `GET` 常用于查询，`POST` 常用于提交或创建。
4. `params` 一般拼在 URL 上，`json` 一般放在请求体里。
5. 因为 HTTP 状态码只表示协议层结果，业务层可能仍然失败。
