## 3. 建议学习方式

这份教材适合按下面顺序学习：

1. 先通读一遍，建立整体概念
2. 第二遍边看边在工具里操作
3. 第三遍对照文档，自己独立调接口
4. 最后完成练习和自测题

不要只看不练。今天最重要的产出不是笔记，而是你亲手调通的接口。

### 3.1 推荐直接练习的公开接口

如果你现在手上没有现成项目接口，建议直接用下面这几套公开接口练习。

#### `JSONPlaceholder`

适合练基础 CRUD、列表、详情、增删改查。

- 首页：`https://jsonplaceholder.typicode.com/`
- Guide：`https://jsonplaceholder.typicode.com/guide`
- 示例：
  - `GET https://jsonplaceholder.typicode.com/posts`
  - `GET https://jsonplaceholder.typicode.com/posts/1`
  - `POST https://jsonplaceholder.typicode.com/posts`

#### `DummyJSON`

适合练更接近真实业务的接口，包括 `products/users/carts/todos`，以及登录鉴权。

- 首页：`https://dummyjson.com/`
- Docs：`https://dummyjson.com/docs`
- Auth Docs：`https://dummyjson.com/docs/auth`
- 示例：
  - `GET https://dummyjson.com/products`
  - `GET https://dummyjson.com/users/1`
  - `POST https://dummyjson.com/auth/login`

#### `Postman Echo`

适合练 `headers/params/body/auth`，它会把你发过去的内容回显回来，非常适合入门。

- Docs：`https://learning.postman.com/docs/developer/echo-api/`
- 示例：
  - `GET https://postman-echo.com/get?foo=bar`
  - `POST https://postman-echo.com/post`

#### `httpbin`

适合练 HTTP 细节，比如状态码、重定向、Basic Auth、Bearer。

- 官网：`https://httpbin.org/`
- 示例：
  - `GET https://httpbin.org/get`
  - `POST https://httpbin.org/post`
  - `GET https://httpbin.org/status/404`
  - `GET https://httpbin.org/basic-auth/user/passwd`

### 3.2 推荐使用顺序

建议按下面顺序练：

1. 先用 `Postman Echo` 和 `httpbin` 练 HTTP 基础
2. 再用 `JSONPlaceholder` 练 CRUD 和查询类接口
3. 最后用 `DummyJSON` 练登录、鉴权和业务链路

### 3.3 没有可用接口时怎么办

如果公开接口不可用、限制太多，或者不适合当前学习目标，后续学习时应该直接补一个本地练习项目，而不是停在“只有理论没有可练环境”。

也就是说：

- 有可用公开接口时，直接用真实接口练
- 没有合适接口时，应该创建本地可运行的练习接口代码，再继续学自动化

这是后续学习材料默认遵循的处理方式。

## 4. 什么是接口

### 4.1 接口的直观理解

在软件系统里，接口可以理解为：

“一个系统对外暴露出来的可调用能力入口。”

例如：

- 登录接口：告诉系统“我要登录”
- 查询用户信息接口：告诉系统“把这个用户资料返回给我”
- 下单接口：告诉系统“帮我生成一笔订单”

接口通常由前端、移动端、其他系统或测试脚本来调用。

### 4.2 一个生活化类比

把系统想象成餐厅后厨：

- 前端页面像服务员
- 用户像顾客
- 后厨像服务系统
- 接口像服务员递给后厨的标准化单据

顾客不直接进后厨炒菜，前端也不直接操作数据库。大家都通过接口沟通。

### 4.3 接口的常见形式

你当前学习的“接口自动化测试”，通常默认是 HTTP 接口测试，也就是 Web API 测试。

常见接口形式有：

- HTTP/HTTPS RESTful API
- RPC 接口
- WebSocket 接口
- GraphQL 接口

第 1 阶段先聚焦 HTTP/HTTPS 接口即可。

## 5. 什么是接口测试

### 5.1 定义

接口测试，就是验证系统接口是否按预期工作。

它关注的不是页面展示，而是：

- 请求有没有正确发送
- 参数有没有被正确处理
- 返回结果是否正确
- 业务逻辑是否成立
- 权限控制是否正确
- 异常输入是否被合理拦截

### 5.2 为什么接口测试重要

接口层处于系统逻辑的核心位置：

- 前端最终调用的是接口
- App 最终调用的是接口
- 系统对系统的通信也常常基于接口

所以接口测得好，很多核心问题能在 UI 层之前被发现。

### 5.3 接口测试和 UI 测试的区别

| 维度 | 接口测试 | UI 测试 |
| --- | --- | --- |
| 关注点 | 数据和逻辑 | 页面和交互 |
| 执行速度 | 更快 | 更慢 |
| 稳定性 | 通常更高 | 容易受页面变化影响 |
| 发现问题位置 | 更接近后端逻辑 | 更接近用户操作 |
| 维护成本 | 相对较低 | 相对较高 |

结论：

接口测试不是替代 UI 测试，而是更高性价比的核心验证手段。

## 6. HTTP 是什么

### 6.1 基本定义

HTTP 是 HyperText Transfer Protocol，即超文本传输协议。

它规定了客户端和服务器如何通信。

你可以简单理解成：

- 客户端发请求
- 服务端收请求并返回响应

### 6.2 一次接口调用发生了什么

当你调用一个接口时，通常会发生下面的过程：

1. 客户端准备请求地址、方法、参数、请求头
2. 请求发送给服务器
3. 服务器接收并处理业务逻辑
4. 服务器返回状态码、响应头、响应体
5. 客户端根据响应内容判断结果

### 6.3 HTTP 与 HTTPS

- HTTP：明文传输
- HTTPS：在 HTTP 基础上加了 SSL/TLS 加密，更安全

现在真实项目里大多数接口都是 HTTPS。

对接口测试来说，第 1 天你只需要知道：

- 二者使用方式接近
- HTTPS 更安全
- 测试时常见地址通常以 `https://` 开头

## 7. URL 是什么

### 7.1 定义

URL 就是请求地址。

例如：

```text
https://api.example.com/users/1001?status=active
```

### 7.2 URL 组成

以上面这个 URL 为例：

- 协议：`https`
- 域名：`api.example.com`
- 路径：`/users/1001`
- 查询参数：`status=active`

### 7.3 你需要重点看什么

做接口测试时，最重要的是看：

- 请求方法是什么
- 资源路径是什么
- 参数在哪里传

---

## 8. HTTP 请求方法

接口文档里最常见的就是请求方法，也叫 HTTP Method。

### 8.1 GET

作用：

- 获取资源
- 常用于查询

典型场景：

- 查询用户信息
- 查询商品列表
- 查询订单详情

示例：

```text
GET /users/1001
GET /products?page=1&page_size=10
```

特点：

- 一般不修改数据
- 参数通常放在 URL 查询串里

### 8.2 POST

作用：

- 新建资源
- 提交数据

典型场景：

- 登录
- 注册
- 创建订单

示例：

```text
POST /login
POST /orders
```

特点：

- 常把参数放在请求体中
- 很多业务操作都用它

### 8.3 PUT

作用：

- 更新资源
- 常表示整体更新

典型场景：

- 更新用户资料
- 修改订单信息

示例：

```text
PUT /users/1001
```

### 8.4 PATCH

作用：

- 局部更新资源

典型场景：

- 只修改用户昵称
- 只修改订单状态

### 8.5 DELETE

作用：

- 删除资源

典型场景：

- 删除用户
- 删除购物车商品

示例：

```text
DELETE /users/1001
```

### 8.6 方法不是绝对业务真理

理论上：

- 查询用 `GET`
- 新增用 `POST`
- 修改用 `PUT/PATCH`
- 删除用 `DELETE`

但真实项目里不一定完全规范。有些系统会：

- 用 `POST` 做查询
- 用 `POST` 做删除

所以你要以接口文档和真实实现为准，不要机械套 REST 理论。

## 9. 请求的组成

一条 HTTP 请求，通常由以下部分组成：

- 请求行
- 请求头 Headers
- 请求体 Body

对于做接口测试的人，最常关注的是：

- URL
- Method
- Headers
- Params
- Body

### 9.1 Headers 是什么

请求头用于描述这次请求的附加信息。

常见请求头：

- `Content-Type`
- `Authorization`
- `Cookie`
- `User-Agent`

示例：

```http
Content-Type: application/json
Authorization: Bearer abcdefg123456
```

常见作用：

- 告诉服务端你发送的数据格式
- 携带身份凭证
- 表示客户端信息

### 9.2 Params 是什么

`params` 通常指 URL 查询参数。

例如：

```text
GET /users?page=1&page_size=20
```

这里：

- `page=1`
- `page_size=20`

就是查询参数。

特点：

- 一般拼在 URL 后面
- 常用于查询场景
- 可读性强

### 9.3 Body 是什么

Body 是请求体，也就是请求真正提交的数据内容。

常见于：

- 登录
- 注册
- 新增数据
- 修改数据

### 9.4 data 和 json 的区别

这是后面用 `requests` 时很容易混淆的点，你今天先建立概念。

#### `data`

通常表示普通表单数据，常见格式：

- `application/x-www-form-urlencoded`
- `multipart/form-data`

例如：

```text
username=tom&password=123456
```

#### `json`

表示 JSON 格式请求体。

例如：

```json
{
  "username": "tom",
  "password": "123456"
}
```

#### 核心区别

- `params`：一般拼在 URL 上
- `data`：一般是表单格式请求体
- `json`：一般是 JSON 格式请求体

不要只记定义，要记“它们分别出现在什么场景”。

## 10. 响应的组成

接口返回结果通常包括：

- 状态码
- 响应头
- 响应体

### 10.1 状态码

状态码表示这次 HTTP 通信在协议层面的大致结果。

例如：

- `200 OK`
- `201 Created`
- `400 Bad Request`
- `401 Unauthorized`
- `403 Forbidden`
- `404 Not Found`
- `500 Internal Server Error`

### 10.2 响应体

响应体通常是 JSON。

例如：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "user_id": 1001,
    "username": "tom"
  }
}
```

你做断言时，很多时候真正看的不是 HTTP 状态码，而是响应体里的业务字段。

### 10.3 重要认知

`HTTP 状态码正常` 不等于 `业务成功`。

例如：

```json
{
  "code": 10001,
  "message": "password error",
  "data": null
}
```

有些系统即使登录失败，也可能仍然返回 `200`，只是业务码不是成功。

所以后面做测试时，一般至少要看两层：

1. HTTP 层是否正确
2. 业务层是否正确

---

## 11. 常见状态码理解

### 11.1 2xx 成功

#### `200`

表示请求成功，最常见。

#### `201`

表示资源创建成功，常见于新增接口。

#### `204`

表示成功，但无响应体。

### 11.2 4xx 客户端错误

表示请求本身有问题。

#### `400`

请求参数错误、格式错误、缺少必填项等。

#### `401`

未认证，常见于：

- 没登录
- Token 无效
- Token 过期

#### `403`

已认证，但没有权限。

例如普通用户访问管理员接口。

#### `404`

资源不存在。

例如接口路径写错，或要查的数据不存在。

### 11.3 5xx 服务端错误

表示服务器内部处理出了问题。

#### `500`

最常见的服务端异常。

做接口测试时，通常你看到 `500`，第一反应不是“测试通过”，而是：

- 服务有 bug
- 需要保留请求参数和响应内容以便排查

## 12. Cookie、Session、Token、JWT

这是 Day 1 非常重要的基础。

### 12.1 为什么需要身份机制

很多接口不是谁都能调用的，比如：

- 查询个人资料
- 提交订单
- 删除数据
- 管理后台操作

系统需要知道“你是谁”，所以要有认证机制。

### 12.2 Cookie

Cookie 是服务端返回给客户端并保存的一小段信息。

后续请求时，客户端会自动带上 Cookie。

可理解为：

- 第一次登录后，服务器给你一个“临时身份纸条”
- 以后你拿着这个纸条来访问

### 12.3 Session

Session 可以理解为服务器端保存的会话信息。

典型流程：

1. 用户登录
2. 服务端创建 Session
3. 服务端把 Session 标识返回给客户端
4. 客户端后续带着这个标识访问

很多传统 Web 系统会使用 Cookie + Session。

### 12.4 Token

Token 是常见的接口认证方式。

典型流程：

1. 调用登录接口
2. 服务端返回一个 token
3. 后续请求把 token 放到请求头里

例如：

```http
Authorization: Bearer eyJhbGciOi...
```

优点：

- 前后端分离系统中很常见
- 使用灵活

### 12.5 JWT

JWT 是 JSON Web Token，是一种特定格式的 Token。

你第 1 天不需要掌握它的底层加密细节，只需要知道：

- 它本质上也是一种身份凭证
- 常见于现代接口系统
- 通常会有过期时间

### 12.6 测试视角下你要关注什么

对于认证机制，测试时重点关注：

- 未登录能不能访问受限接口
- 登录后能不能正常访问
- token 错误时是否被拦截
- token 过期时是否被拦截
- 普通用户是否越权访问管理员接口

## 14. 如何手工调用一个接口

今天你至少要会用一种工具手工调接口。推荐：

- Postman
- Apifox

二选一即可。

### 14.1 手工调用接口的标准步骤

1. 确认请求方法
2. 输入请求 URL
3. 配置请求头
4. 配置查询参数或请求体
5. 点击发送
6. 查看状态码
7. 查看响应体
8. 判断结果是否符合预期

### 14.2 以登录接口为例

假设登录接口文档如下：

- URL：`https://api.example.com/api/v1/login`
- Method：`POST`
- Headers：`Content-Type: application/json`
- Body：

```json
{
  "username": "tom",
  "password": "123456"
}
```

成功响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "abc123"
  }
}
```

你在工具里应该这样操作：

1. 选择 `POST`
2. 输入 URL
3. 在 Headers 中加 `Content-Type: application/json`
4. 在 Body 中选择 `raw` + `JSON`
5. 输入登录 JSON
6. 点击 Send
7. 看返回中是否有 token

### 14.3 以查询用户详情接口为例

接口文档：

- URL：`https://api.example.com/api/v1/users/1001`
- Method：`GET`
- Headers：`Authorization: Bearer abc123`

你在工具里应该这样操作：

1. 选择 `GET`
2. 输入 URL
3. 在 Headers 中加入 token
4. 点击发送
5. 检查状态码和用户信息是否正确

## 15. 接口测试到底测什么

这是今天最关键的部分之一。

很多初学者只会“接口通了就算通过”，这是不够的。测试要验证的是“系统是否符合预期”。

通常至少从下面几个维度考虑。

### 15.1 功能正确性

最基础的问题：

- 接口能不能成功调用
- 返回结果是不是预期结果

例如登录接口：

- 正确用户名 + 正确密码，应该登录成功

### 15.2 参数校验

检查系统是否正确处理输入参数。

常见测试点：

- 必填项缺失
- 参数类型错误
- 参数长度超限
- 参数格式错误
- 参数取值越界

例如注册接口：

- 用户名为空
- 密码太短
- 邮箱格式错误

### 15.3 业务规则校验

不是所有问题都能从参数合法性判断，有些是业务规则问题。

例如：

- 已注册手机号不能重复注册
- 库存不足时不能下单
- 已删除订单不能再次支付

### 15.4 鉴权校验

判断接口访问权限是否正确。

常见测试点：

- 未登录访问受限接口
- token 错误
- token 过期
- 普通用户访问管理员接口

### 15.5 异常场景校验

测试系统在非正常输入下是否仍然稳定。

例如：

- 传空值
- 传超长字符串
- 传特殊字符
- 传不存在的 ID

### 15.6 数据一致性校验

尤其是增删改接口，要验证“结果真的生效了”。

例如：

- 创建用户后，查询接口能查到
- 删除用户后，查询接口查不到

这类测试思路会在后面的接口关联和数据库断言中继续展开。

## 17. 登录接口测试点示例

下面给你一个完整的思路示范。

### 17.1 接口描述

接口名称：用户登录

请求方式：`POST`

接口地址：

```text
/api/v1/login
```

请求体：

```json
{
  "username": "tom",
  "password": "123456"
}
```

### 17.2 正向测试点

- 正确用户名 + 正确密码，登录成功
- 成功返回 token
- 返回用户基本信息

### 17.3 反向测试点

- 用户名为空
- 密码为空
- 用户名不存在
- 密码错误
- 用户名长度超限
- 密码长度超限
- 用户名包含非法字符
- 请求体缺少字段

### 17.4 鉴权和安全相关测试点

- 多次错误登录是否有限制
- 被禁用账号是否无法登录
- 登录成功后 token 是否有效
- 不同角色登录返回权限是否正确

### 17.5 协议层测试点

- 请求方法错误，例如用 `GET` 调登录接口
- `Content-Type` 错误
- 请求体格式不是 JSON

接口名称：商品列表

请求方式：`GET`

示例：

```text
GET /api/v1/products?page=1&page_size=10
```

### 18.1 正向测试点

- 默认分页查询成功

- `page=-1`
- `page_size=0`
- `page_size` 超过最大限制

### 18.3 业务测试点

- 查询结果是否按预期排序
- 是否正确返回筛选条件结果
- 总数和列表项是否一致

### 18.4 边界场景

- 最后一页
- 超出总页数
- 无数据场景

## 21. 接口测试中常见误区

### 21.1 误区一：返回 200 就算通过

不对。

很多系统业务失败也可能返回 `200`，只是业务码不同。

### 21.2 误区二：只测正常场景

不对。

异常参数、边界值、权限问题往往更容易暴露 bug。

### 21.3 误区三：只看响应，不看请求

不对。

定位问题时，必须同时保留：

- 请求地址
- 请求头
- 请求参数
- 响应结果

### 21.4 误区四：不关注业务规则

不对。

接口测试不是“纯协议测试”，业务逻辑才是重点。

### 21.5 误区五：文档不一致就直接放弃

不对。

真实项目里文档不一定总是完全准确，你要学会：

- 用工具调试
- 和开发确认
- 以真实返回和业务规则为准

## 25. 今日练习题参考答案

### 练习 1 参考答案

- URL：请求地址
- Header：请求附加信息，例如数据格式、身份信息
- Body：请求体，提交的数据内容
- Token：身份凭证
- Status Code：HTTP 协议层面的返回状态

### 练习 2 参考答案

- 查询用户详情：`GET`
- 创建订单：`POST`
- 删除购物车商品：`DELETE`
- 更新用户昵称：`PUT` 或 `PATCH`

### 练习 3 参考答案

- HTTP 层面：成功，因为返回 `200`
- 业务层面：失败，因为业务码显示密码错误

### 练习 4 参考答案

至少包括：

- 登录后正常访问
- 未登录访问
- token 无效或过期后访问

### 练习 5 参考答案

- `params`：列表查询、详情查询、搜索筛选
- `json`：登录、注册、新增、修改
