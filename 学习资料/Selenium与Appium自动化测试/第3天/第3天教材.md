## 5. 为什么 Day 3 会开始“脚本不稳定”

前两天你练的大多是：

- 页面一打开，元素就已经在
- 输入框和按钮都是立即可操作的

但真实项目里常常不是这样。

常见情况包括：

- 页面接口还没返回，内容还没渲染出来
- 按钮虽然已经出现在 DOM 里，但还没真正可点击
- 页面局部刷新后，你手里拿着的是旧元素引用
- 覆盖层还没消失，按钮被挡住了

也就是说：

很多“不稳定”不是 Selenium 随机抽风，而是页面状态还没准备好。

## 6. Day 3 先学会判断问题来源

你看到报错时，先不要急着加 `sleep()`。

先判断它更像哪类问题：

1. 元素根本还没出现
2. 元素出现了，但不可见
3. 元素可见了，但还不能点
4. 元素已经被页面重绘替换掉了
5. 你点的按钮被别的层挡住了

拿 Day 3 页面举例：

- “开始动态加载”对应：内容延迟出现
- “3 秒后启用按钮”对应：元素存在但暂时不可用
- “重渲染列表”对应：旧元素引用失效
- “覆盖层观察区”对应：按钮被挡住

### 6.1 随堂检查

1. 页面上能看到按钮，是否一定代表 Selenium 现在就能点到它？
2. 脚本时好时坏，最常见是代码随机错了吗？
3. 如果页面重渲染了，旧元素引用为什么可能失效？

### 6.2 答案

1. 不一定，按钮可能还不可点击，或者被覆盖层挡住。
2. 通常不是，更常见是页面状态和脚本执行时机没对上。
3. 因为页面重渲染后，原来的 DOM 节点已经不是当前页面里的有效节点了。

## 7. 显式等待是什么

显式等待，就是：

你明确告诉脚本“等某个条件成立，再继续往下执行”。

常见写法：

```python
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

wait = WebDriverWait(driver, 10)
wait.until(EC.visibility_of_element_located((By.ID, "delayed-message")))
```

它比盲目 `sleep()` 更好的原因是：

- 条件一满足就继续，不会死等
- 条件不满足时会给出更明确的超时报错
- 更接近真实页面状态

### 7.1 Day 3 最常用的条件

```python
EC.presence_of_element_located(...)
EC.visibility_of_element_located(...)
EC.element_to_be_clickable(...)
EC.text_to_be_present_in_element(...)
```

你现在先这样理解：

- `presence`：元素进 DOM 了
- `visibility`：元素看得见了
- `clickable`：元素能点击了
- `text_to_be_present`：某段文字真的出现了

### 7.2 动态内容等待示例

```python
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

driver.find_element(By.ID, "start-loading").click()
wait = WebDriverWait(driver, 10)
wait.until(
    EC.text_to_be_present_in_element((By.ID, "delayed-message"), "Hello Day 3")
)
print(driver.find_element(By.ID, "delayed-message").text)
```

### 7.3 随堂检查

1. 为什么显式等待通常比 `time.sleep()` 更适合真实项目？
2. `presence_of_element_located` 和 `visibility_of_element_located` 一样吗？
3. 如果你要等按钮真正能点，优先想到哪个条件？

### 7.4 答案

1. 因为显式等待会等“条件”，不是傻等固定时间，通常更稳也更高效。
2. 不一样。前者只表示元素进了 DOM，后者还要求元素可见。
3. 优先想到 `element_to_be_clickable`。

## 8. 隐式等待和显式等待的区别

隐式等待的意思是：

给整个 driver 设置一个全局查找等待时间。

例如：

```python
driver.implicitly_wait(5)
```

它的特点是：

- 对后续 `find_element` 生效
- 找到就立刻返回
- 找不到就最多等指定时间

但 Day 3 更推荐你优先掌握显式等待，因为显式等待更精确。

你可以先记这个原则：

- Day 3 学习阶段：优先理解显式等待
- 隐式等待了解概念即可
- 不要把所有问题都丢给隐式等待

### 8.1 随堂检查

1. 隐式等待更像全局规则，还是单点条件？
2. Day 3 为什么更推荐先掌握显式等待？

### 8.2 答案

1. 更像全局规则。
2. 因为显式等待更精确，能明确表达“到底在等什么条件成立”。

## 9. 断言是什么，Day 3 为什么开始要写断言

如果没有断言，你只能知道：

- 代码有没有报错

但你不知道：

- 页面结果到底对不对
- 动态内容是不是真的出现了
- 按钮点击后的结果是不是你要的

Day 3 最基础的断言包括：

```python
assert "Hello Day 3" in driver.find_element(By.ID, "delayed-message").text
assert driver.find_element(By.ID, "delayed-action").is_enabled()
assert driver.find_element(By.ID, "action-result").get_attribute("data-clicked") == "true"
```

你可以先记一句：

`等待负责让页面准备好，断言负责确认结果是对的。`

### 9.1 随堂检查

1. 没有断言时，你最多知道什么？
2. 为什么等待和断言不是一回事？

### 9.2 答案

1. 最多知道代码有没有报错。
2. 因为等待是在等页面状态，断言是在判断结果是否符合预期。

## 10. Day 3 常见异常怎么理解

Day 3 不要求你把所有异常都背下来，但要先认得几个高频名字。

### 10.1 `NoSuchElementException`

更像是在说：

“我现在找不到这个元素。”

常见原因：

- 定位写错
- 页面还没加载出来
- 元素在别的 frame 里

### 10.2 `TimeoutException`

更像是在说：

“我等了，但到超时都没等到条件成立。”

常见原因：

- 页面真的没出现目标状态
- 等待条件写错
- 超时时间太短

### 10.3 `StaleElementReferenceException`

更像是在说：

“你拿的是旧元素引用，它已经不是当前页面上的有效节点了。”

Day 3 页面里的“重渲染列表”就是典型场景。

### 10.4 `ElementClickInterceptedException`

更像是在说：

“你要点的元素被别的东西挡住了。”

覆盖层、弹窗、遮罩层都可能导致这个问题。

### 10.5 覆盖层场景怎么处理

如果你知道页面上有遮罩层、loading 层或新手引导层，就不要急着硬点目标按钮。

更稳的思路是：

1. 先确认是不是被遮挡
2. 再等待覆盖层消失
3. 最后再点击目标按钮

示例：

```python
from selenium.common.exceptions import ElementClickInterceptedException
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

button = driver.find_element(By.ID, "covered-button")

try:
    button.click()
except ElementClickInterceptedException:
    wait = WebDriverWait(driver, 10)
    wait.until(EC.invisibility_of_element_located((By.ID, "blocking-overlay")))
    driver.find_element(By.ID, "covered-button").click()
```

这里有两个细节：

- 不要把“被挡住”误判成“定位错了”
- 等覆盖层消失后，最好重新定位一次按钮再点击

### 10.6 异常名不要死背，先学会翻译

你看到异常时，先把它翻译成一句人话：

- 找不到
- 等不到
- 旧了
- 被挡了

这样你排查速度会快很多。

### 10.7 随堂检查

1. `TimeoutException` 更像“没找到”，还是“等了但没等到”？
2. `StaleElementReferenceException` 的核心原因是什么？
3. `ElementClickInterceptedException` 最常见是在提示什么？

### 10.8 答案

1. 更像“等了但没等到”。
2. 你持有的是旧元素引用，页面已经重绘或替换了节点。
3. 通常是在提示目标元素被别的层挡住了。

### 12.5 练习参考答案

基础练习参考代码：

```python

from selenium.webdriver.common.by import By

driver = webdriver.Edge()
driver.get("http://127.0.0.1:8000/pages/day3.html")

driver.find_element(By.ID, "start-loading").click()

assert "Hello Day 3" in message
print(message)

driver.quit()
```

加强练习参考代码：

```python
from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

driver = webdriver.Edge()
driver.get("http://127.0.0.1:8000/pages/day3.html")

driver.find_element(By.ID, "enable-action").click()
wait = WebDriverWait(driver, 10)
button = wait.until(EC.element_to_be_clickable((By.ID, "delayed-action")))
button.click()

result = driver.find_element(By.ID, "action-result")
assert result.get_attribute("data-clicked") == "true"
print(result.text)

driver.quit()
```

异常观察参考思路：

1. 先保存旧元素引用
2. 页面重渲染后，不要继续相信这个旧引用
3. 如果报 `StaleElementReferenceException`，重新定位

## 14. 自测题

### 14.1 概念题

1. 为什么很多 Selenium 问题本质上是“时机问题”？
2. 显式等待和断言分别负责什么？
3. `StaleElementReferenceException` 的根本原因是什么？

### 14.2 代码理解题

看下面代码，回答问题：

```python
from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

driver = webdriver.Edge()
driver.get("http://127.0.0.1:8000/pages/day3.html")

driver.find_element(By.ID, "enable-action").click()
wait = WebDriverWait(driver, 10)
button = wait.until(EC.element_to_be_clickable((By.ID, "delayed-action")))
button.click()
print(driver.find_element(By.ID, "action-result").text)

driver.quit()
```

请回答：

1. 这里为什么不能直接点 `delayed-action`？
2. `element_to_be_clickable` 在等什么？
3. 这段代码执行成功后，页面上应该发生什么？

### 14.3 标准答案

1. 因为按钮一开始是禁用状态，直接点很可能失败。
2. 它在等待元素真正可点击，也就是可见且可操作。
3. 按钮被成功点击，`action-result` 会变成“延迟按钮点击成功。”。

### 14.4 概念题参考答案

1. 因为很多报错并不是定位本身错，而是页面状态还没准备好。
2. 显式等待负责等条件成立，断言负责验证结果是否正确。
3. 因为页面重渲染后，旧元素引用已经不再指向当前有效节点。
