## 5. 平台特有定位怎么理解

Day 5 你已经知道：

- `ACCESSIBILITY_ID`
- `ID`
- `XPATH`

Day 6 要补上平台特有定位。

### 5.1 Android：`ANDROID_UIAUTOMATOR`

这个定位最常见的价值是：

- 结合 Android 原生 UIAutomator 语法
- 做滚动查找
- 做更偏 Android 原生的元素筛选

典型写法：

```python
driver.find_element(
    AppiumBy.ANDROID_UIAUTOMATOR,
    'new UiScrollable(new UiSelector().scrollable(true)).scrollIntoView(text("Views"))'
)
```

你现在先不用背整串语法，只要记住它最常用的用途：

`Android 上，滚动到某个文本元素。`

### 5.2 iOS：`IOS_PREDICATE`

这是 iOS 上非常常见的稳定定位方式。

典型写法：

```python
driver.find_element(
    AppiumBy.IOS_PREDICATE,
    'label == "Views"'
)
```

你可以把它理解成：

`用属性表达式筛选 iOS 元素。`

### 5.3 iOS：`IOS_CLASS_CHAIN`

它更像是 iOS 上的一种层级筛选方式，比 XPath 更轻量一些。

典型写法：

```python
driver.find_element(
    AppiumBy.IOS_CLASS_CHAIN,
    '**/XCUIElementTypeCell[`label == "Views"`]'
)
```

Day 6 不要求你立刻默写它，但你要知道：

- Android 有 `ANDROID_UIAUTOMATOR`
- iOS 有 `IOS_PREDICATE` / `IOS_CLASS_CHAIN`

### 5.4 随堂检查

1. Android 平台特有定位里，最常见的用途是什么？
2. iOS 为什么常常更推荐 `IOS_PREDICATE`，而不是一上来就 XPath？

### 5.5 答案

1. 滚动查找和 Android 原生规则筛选。
2. 因为通常更轻、更清晰，也比长 XPath 更稳定。

## 6. 移动端为什么经常要做滚动和手势

Web 页面很多元素一进页面就能看到。

移动端不一样，常见情况是：

- 屏幕可视区域更小
- 列表内容更长
- 需要滚动才能看到目标元素
- 需要长按、滑动、拖动才能触发交互

所以 Day 6 的一个核心思维是：

`移动端定位常常不是“定位不到”，而是“元素还没滚到视野里”。`

---

## 7. Day 6 的手势先怎么学

今天先按“从稳到复杂”的顺序理解。

### 7.1 点击仍然优先用 click

能直接：

```python
element.click()
```

就不要一上来就写复杂手势。

### 7.2 滚动优先区分“滚动查找”还是“坐标滑动”

如果目标是：

- 找到一个屏幕外的元素

优先考虑：

- Android：`ANDROID_UIAUTOMATOR` 的 `UiScrollable`

如果目标是：

- 真的模拟一个手指滑动动作

再考虑更底层的手势。

### 7.3 长按怎么理解

长按通常不是“普通 click 延长版”，而是会触发不同交互：

- 弹菜单
- 进入编辑模式
- 多选模式

在 Appium 2 时代，很多旧式 `TouchAction` 教程已经不适合作为主线记忆了。Day 6 更建议你建立这个原则：

`优先使用当前客户端支持的 W3C Actions 或驱动支持的 mobile 命令，不要死记老旧 TouchAction。`

### 7.4 滑动和滚动别混成一件事

你可以先粗略这样区分：

- 滚动：为了让内容进入可视区
- 滑动：更强调一个手势动作本身

学习阶段最常见误区是：

- 明明只想找屏幕下方元素
- 却上来就写复杂坐标滑动

这会把问题变复杂。

### 7.5 随堂检查

1. Day 6 为什么不建议一上来就把所有移动交互都写成复杂手势？
2. 如果目标只是让元素滚进可视区，优先想到什么？

### 7.6 答案

1. 因为很多场景普通 click 或滚动查找已经够用，复杂手势会增加不稳定性。
2. 优先想到滚动查找，例如 Android 的 `UiScrollable`。

## 8. 权限弹窗、Toast、系统弹窗

这是 Day 6 必须知道的移动端特殊场景。

### 8.1 权限弹窗

常见权限：

- 相机
- 相册
- 定位
- 通知

这类弹窗的关键不是“代码会不会点”，而是：

- Android 和 iOS 的按钮文本不同
- 不同系统版本文案可能不同
- 首次安装和已授权状态不同

所以 Day 6 要建立这个思路：

`权限弹窗不是写死一个按钮文本就结束，而是要考虑状态分支。`

### 8.2 Toast

Toast 的特点是：

- 出现快
- 消失快
- 不是普通静态文本

所以处理 Toast 的关键通常是：

- 快速等待
- 不要指望它一直留在页面上

### 8.3 系统弹窗

系统层弹窗往往意味着：

- 不是业务页面本身的一部分
- 不能完全按普通页面流程想

所以 Day 6 很重要的一句话是：

`移动端自动化要分清“App 页面”和“系统层界面”。`

## 9. WebView 和 Native Context

这是 Appium 和 Selenium 开始“真正接轨”的地方。

### 9.1 什么是 Context

在移动端里，常见两类上下文：

- `NATIVE_APP`
- `WEBVIEW_xxx`

当 App 是纯原生页面时，你通常在：

```python
NATIVE_APP
```

当 App 某一页嵌了 WebView 时，你可能需要切到：

```python
WEBVIEW_xxx
```

### 9.2 为什么这很重要

因为：

- 原生页面定位方式和 Web 页面不同
- 你不切 context，可能就找不到元素

### 9.3 最小理解代码

```python
print(driver.contexts)
driver.switch_to.context("WEBVIEW_com.demo")
print(driver.current_context)
driver.switch_to.context("NATIVE_APP")
```

你现在先不用背它，只要先知道：

`Hybrid App 要先确认自己在哪个上下文里。`

### 9.4 随堂检查

1. 为什么 Hybrid App 里会出现“明明看见元素却找不到”？
2. `NATIVE_APP` 和 `WEBVIEW_xxx` 的本质区别是什么？

### 9.5 答案

1. 很可能是因为当前 context 不对。
2. 前者是原生页面上下文，后者是 WebView 页面上下文。

## 10. Appium + pytest 该怎么接

这块是 Day 6 最关键的项目化能力。

你会发现它和 Day 4 非常像，只是浏览器换成了移动端 driver。

### 10.1 最小 fixture

```python
import pytest
from appium import webdriver
from appium.options.android import UiAutomator2Options


@pytest.fixture
def mobile_driver():
    options = UiAutomator2Options()
    options.platform_name = "Android"
    options.automation_name = "UiAutomator2"
    options.device_name = "Android Emulator"
    options.app_package = "io.appium.android.apis"
    options.app_activity = ".ApiDemos"

    driver = webdriver.Remote("http://127.0.0.1:4723", options=options)
    yield driver
    driver.quit()
```

### 10.2 这段和 Day 4 的共通点

- 一样是 `fixture`
- 一样是 `yield` 前准备资源
- 一样是 `yield` 后回收资源

区别只是：

- Day 4 准备的是浏览器
- Day 6 准备的是移动端会话

### 10.3 测试函数怎么写

```python
def test_open_views(mobile_driver):
    wait = WebDriverWait(mobile_driver, 10)
    views = wait.until(
        EC.presence_of_element_located((AppiumBy.ACCESSIBILITY_ID, "Views"))
    )
    views.click()

    animation = wait.until(
        EC.presence_of_element_located((AppiumBy.ACCESSIBILITY_ID, "Animation"))
    )
    assert animation.is_displayed()
```

你可以看到：

- `fixture` 提供 driver
- 测试函数负责流程和断言

这和 Day 4 的结构是完全共通的。

## 11. 移动端 POM 怎么迁移

Day 6 不要求你把移动端框架做得很重，但你要理解：

`POM` 在移动端也成立。`

### 11.1 移动端页面对象负责什么

仍然是：

- 定位
- 动作
- 读取结果

例如：

```python
class HomePage:
    def __init__(self, driver):
        self.driver = driver

    def open_views(self):
        self.driver.find_element(AppiumBy.ACCESSIBILITY_ID, "Views").click()

    def has_animation_entry(self):
        return self.driver.find_element(AppiumBy.ACCESSIBILITY_ID, "Animation").is_displayed()
```

### 11.2 测试函数仍然负责什么

- 组织场景
- 等待
- 断言

也就是：

```python
def test_views_page(mobile_driver):
    page = HomePage(mobile_driver)
    page.open_views()
    assert page.has_animation_entry()
```

### 11.3 一句迁移口诀

`Web 的 POM 思维不变，变的是定位方式和设备环境。`

## 12. Web 自动化和移动端自动化怎么统一设计

这是 Day 6 的收尾重点。

你现在要开始建立统一视角：

### 12.1 共通的东西

- `driver`
- 定位
- 等待
- 断言
- `pytest`
- `fixture`
- `POM`

### 12.2 不同的东西

- 环境链路
- 启动方式
- 平台驱动
- 定位策略
- 权限/Toast/WebView/上下文
- 手势

### 12.3 真正统一的不是语法，而是设计思想

真正统一的是：

- 资源统一交给 fixture
- 页面动作收进页面对象
- 测试函数负责业务流程和断言
- 等待优先围绕业务结果

如果你抓住的是这套思想，那你以后切换：

- Selenium
- Appium
- Playwright

都会更快。

### 16.2 代码理解题

看下面代码，回答问题：

```python
@pytest.fixture
def mobile_driver():
    options = UiAutomator2Options()
    options.platform_name = "Android"
    options.automation_name = "UiAutomator2"
    options.device_name = "Android Emulator"

    driver = webdriver.Remote("http://127.0.0.1:4723", options=options)
```

请回答：

1. 这段 fixture 负责什么？
