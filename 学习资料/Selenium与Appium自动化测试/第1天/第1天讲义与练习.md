## 4. 重点一：UI 自动化测试到底是什么

### 4.1 一句话理解

UI 自动化测试，就是让脚本代替人去操作页面，并检查结果是否符合预期。

### 4.2 Day 1 先记住这几个关键词

- 打开页面
- 找到元素
- 执行动作
- 获取结果
- 判断是否正确

### 4.3 它和接口测试的关系

你已经有接口测试基础，所以这里直接建立映射：

- 接口测试：直接调服务
- UI 自动化：从页面入口走完整链路

接口测试更快，UI 自动化更接近用户真实操作。

### 4.4 随堂检查

1. UI 自动化最核心的价值是什么？
2. 为什么它不能完全替代接口测试？

### 4.5 答案方向

1. 验证用户真实操作链路是否可用。
2. 因为它更慢、更脆弱，也不如接口测试那样直接验证后端逻辑。

---

## 5. 重点二：Selenium 在做什么

### 5.1 一句话理解

Selenium 让 Python 脚本具备“控制浏览器”的能力。

### 5.2 三层关系一定要记住

- 你的 Python 脚本发命令
- 浏览器驱动负责转发命令
- 浏览器执行命令

记忆公式：

`脚本 -> 驱动 -> 浏览器`

### 5.3 初学时不要陷进去的点

Day 1 不要把注意力放在：

- WebDriver 协议细节
- 各种高级定位技巧
- 框架封装

今天只抓“最小闭环”。

### 5.4 随堂检查

1. Selenium 为什么不能跳过 driver 直接控制浏览器？
2. 脚本、driver、浏览器这三者分别负责什么？

### 5.5 答案方向

1. 因为脚本不能直接控制浏览器内部行为，需要 driver 负责转发和翻译命令。
2. 脚本负责发出自动化指令，driver 负责中转和翻译，浏览器负责真正执行这些动作。

## 7. 重点四：第一个脚本只学最关键的 7 个点

### 7.1 完整代码

```python
from selenium import webdriver
from selenium.webdriver.common.by import By

driver = webdriver.Chrome()
driver.get("http://127.0.0.1:8000/pages/day1.html")

driver.find_element(By.NAME, "user_name").send_keys("hello selenium")
driver.find_element(By.NAME, "study_role").send_keys("Day 1 学习者")
driver.find_element(By.ID, "day1-submit").click()

message = driver.find_element(By.ID, "day1-message").text
print(message)

driver.quit()
```

### 7.2 今天只吃透这 7 个点

1. `webdriver.Chrome()`
2. `driver.get()`
3. `find_element()`
4. `By.NAME`
5. `send_keys()`
6. `click()`
7. `.text` 和 `driver.quit()`

### 7.3 代码执行顺序

1. 启动浏览器
2. 打开页面
3. 找输入框并输入
4. 找按钮并点击
5. 找结果并读取
6. 关闭浏览器

你要先理解顺序，再记代码。

建议你今天至少做一次：

1. 先运行 `01_环境验证.py`
2. 再运行 `02_第一个Selenium脚本.py`
3. 最后自己不看代码重写一遍

### 7.4 随堂检查

1. `find_element()` 是做什么的？
2. `send_keys()` 和 `click()` 分别对应什么用户动作？
3. `.text` 是获取什么？
4. 为什么最后一定要 `quit()`？

### 7.5 答案方向

1. `find_element()` 用来在页面中查找一个元素。
2. `send_keys()` 对应输入，`click()` 对应点击。
3. `.text` 用来获取元素中显示给用户看的文本。
4. 因为 `quit()` 会关闭浏览器并结束当前会话，不写容易留下残余进程。
