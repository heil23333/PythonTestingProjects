## 5. 什么是 UI 自动化测试

### 5.1 直观理解

UI 自动化测试，就是让程序代替人去操作页面，并验证页面行为是否符合预期。

例如一个人平时会做这些动作：

- 打开网页
- 输入用户名和密码
- 点击登录按钮
- 查看提示信息

UI 自动化做的也是同样的事，只不过执行这些动作的不是人，而是脚本。

### 5.2 它到底在测什么

UI 自动化通常关注：

- 页面元素能不能正常显示
- 页面交互能不能正常执行
- 用户操作流程能不能走通
- 页面结果是否符合预期

它不只是“能不能点一下按钮”，而是要验证：

- 这个按钮是否可点击
- 点击之后页面是否按预期变化
- 输入的数据是否被正确处理
- 页面展示结果是否正确

### 5.3 UI 自动化和接口测试的区别

你已经学过 `requests` 和 `pytest`，所以这里很容易建立对比。

| 维度 | UI 自动化测试 | 接口测试 |
| --- | --- | --- |
| 关注点 | 页面和交互 | 数据和业务逻辑 |
| 执行速度 | 相对更慢 | 更快 |
| 稳定性 | 更容易受页面变化影响 | 通常更稳定 |
| 定位问题 | 更接近用户视角 | 更接近后端逻辑 |
| 适用场景 | 核心业务流程回归 | 接口逻辑验证、数据校验 |

你可以这样理解：

- 接口测试更像“直接和后端对话”
- UI 自动化更像“站在用户角度操作系统”

它们不是替代关系，而是互补关系。

### 5.4 为什么 UI 自动化还值得学

虽然 UI 自动化比接口测试慢、维护成本更高，但它依然很重要，因为它能验证：

- 页面能不能真正让用户完成操作
- 前后端联调后的完整链路是否正常
- 关键流程是否能在真实操作层面跑通

比如：

- 登录
- 搜索
- 提交表单
- 下单
- 上传文件

这些都是很典型的 UI 自动化场景。

## 6. Selenium 是什么

### 6.1 定义

Selenium 是一个用于 Web 自动化的工具库。

## 7. Selenium 是怎么工作的

### 7.1 三个核心角色

你可以先把整个链路理解成三层：

1. 自动化脚本
2. 浏览器驱动
3. 浏览器

用更直白的话说：

- Python 脚本发指令
- 驱动负责翻译和转发指令
- 浏览器执行这些指令

### 7.2 一个生活化类比

可以把它理解成：

- 你自己：测试脚本
- 翻译：浏览器驱动
- 外国服务员：浏览器

你不会直接和浏览器内部沟通，而是通过驱动这个中间层来完成控制。

### 7.3 为什么需要驱动

因为脚本不能直接操作浏览器内部行为。

驱动的作用就是：

- 接收来自 Selenium 的命令
- 按浏览器能理解的方式执行这些命令
- 再把执行结果返回给脚本

### 7.4 Day 1 先建立这个模型就够了

今天你不需要深挖 WebDriver 协议细节。

只需要把这条链路记住：

`Python 脚本 -> Selenium -> 浏览器驱动 -> 浏览器`

后面讲定位、等待、异常时，你会反复用到这个理解。

## 9. 第一个 Selenium 脚本

### 9.1 先看完整代码

```python
from selenium import webdriver
from selenium.webdriver.common.by import By

driver = webdriver.Chrome()
driver.get("http://127.0.0.1:8000/pages/day1.html")

driver.find_element(By.NAME, "user_name").send_keys("hello selenium")
driver.find_element(By.NAME, "study_role").send_keys("Day 1 学习者")
driver.find_element(By.ID, "day1-submit").click()

message = driver.find_element(By.ID, "day1-message").text
print(message)

driver.quit()
```

### 9.2 这段代码做了什么

它完成了一个最小自动化流程：

1. 启动浏览器
2. 打开页面
3. 找到输入框
4. 输入文本
5. 找到按钮并点击
6. 找到结果区域并读取文本
7. 关闭浏览器

这就是 Day 1 最核心的闭环。

对应的配套文件是：

- `示例代码/02_第一个Selenium脚本.py`

### 9.3 逐行解释

#### `from selenium import webdriver`

导入浏览器驱动入口。

你后面无论用 Chrome、Edge，都会从这里创建 driver。

#### `from selenium.webdriver.common.by import By`

导入定位方式枚举。

今天你只需要先认识它，不需要一次把所有定位方式都记全。

Day 1 先接触这几个就够了：

- `By.NAME`
- `By.TAG_NAME`
- `By.ID`

#### `driver = webdriver.Chrome()`

创建一个 Chrome 浏览器实例。

从这一行开始，你的脚本就拥有了一个可操作的浏览器对象。

#### `driver.get(...)`

打开目标页面。

这一行和你手动在浏览器地址栏里输入网址的作用是一样的。

#### `driver.find_element(By.NAME, "user_name").send_keys("hello selenium")`

这一行做了两件事：

1. 找到输入框
2. 往输入框里输入文本

这里先不要急着纠结定位方式为什么这样写，Day 2 会专门讲定位。

今天你只要知道：

- `find_element` 是找元素
- `send_keys` 是输入内容

#### `driver.find_element(By.ID, "day1-submit").click()`

这一行同样做了两件事：

1. 找到按钮
2. 点击按钮

#### `message = driver.find_element(By.ID, "day1-message").text`

找到结果区域，并通过 `.text` 读取页面文本。

#### `print(message)`

把获取到的结果打印出来，方便你确认脚本有没有真正执行成功。

#### `driver.quit()`

关闭浏览器并结束会话。

这个动作很重要。

如果少了它，浏览器进程可能不会干净地退出。

## 11. 一个完整动作链路是怎么串起来的

Day 1 最重要的不是背 API，而是建立动作链路理解。

一个最小 Selenium 脚本，通常按下面顺序执行：

1. 创建 driver
2. 打开页面
3. 找元素
4. 对元素执行操作
5. 获取结果
6. 关闭 driver

你可以先把所有 Web 自动化都理解成这 6 步的不同组合。

后面学的定位、等待、断言、pytest、POM，本质上都是在强化这条主线。

## 12. 运行结果怎么看

今天你不能只看“代码有没有报错”，还要看结果是不是符合预期。

至少要确认：

- 浏览器有没有成功打开
- 页面有没有真的被访问
- 输入框有没有被输入内容
- 按钮有没有被点击
- 页面结果文本有没有被正确获取
- 脚本结束后浏览器有没有正确关闭

如果控制台里打印出了页面返回结果，同时浏览器没有异常卡死，说明 Day 1 基本跑通了。

## 13. Day 1 常见问题

### 13.1 浏览器起不来

常见原因：

- 浏览器未安装
- 驱动环境异常
- 公司网络阻止驱动下载
- Python 环境混乱

### 13.2 页面打开了，但找不到元素

Day 1 常见原因通常不是复杂等待问题，而是：

- 页面没开对
- 元素定位写错
- 复制了错误的属性值

### 13.3 能输入，但点击后结果不符合预期

优先检查：

- 你点到的是不是正确按钮
- 页面是否真的跳转或提交
- 结果区域的定位是不是写错了

### 13.4 忘记关闭浏览器

如果你发现每次运行后都留下一堆浏览器窗口，先检查是不是漏写了：

```python

```

## 16. 自测题

### 16.1 概念题

1. UI 自动化测试主要是在验证什么？
2. Selenium 在自动化链路中扮演什么角色？
3. 为什么脚本不能直接跳过 driver 去操作浏览器？

### 16.2 代码理解题

看下面代码，回答问题：

```python
from selenium import webdriver
from selenium.webdriver.common.by import By

driver = webdriver.Chrome()
driver.get("http://127.0.0.1:8000/pages/day1.html")
driver.find_element(By.NAME, "user_name").send_keys("test")
driver.find_element(By.NAME, "study_role").send_keys("Day 1 学习者")
driver.find_element(By.ID, "day1-submit").click()
print(driver.find_element(By.ID, "day1-message").text)
driver.quit()
```

请回答：

1. 这段代码依次做了哪些事？
2. `By.NAME` 和 `By.ID` 是干什么的？
3. `.text` 是做什么的？
4. `driver.quit()` 为什么不能随便删？

### 16.3 标准答案方向

1. 启动浏览器、打开页面、定位输入框、输入内容、点击按钮、获取结果、关闭浏览器。
2. 它们是元素定位方式。
3. 用来获取页面元素中的文本内容。
4. 因为它负责结束会话和关闭浏览器，不写容易导致进程残留。

### 16.4 概念题参考答案

1. UI 自动化测试主要验证页面和交互流程是否符合预期，例如元素是否可操作、流程是否能走通、结果是否正确。
2. Selenium 的作用是让脚本具备控制浏览器的能力，它负责把你的自动化操作传给浏览器执行。
3. 因为脚本本身不能直接操作浏览器内部行为，需要通过浏览器驱动这个中间层来翻译和转发命令。
