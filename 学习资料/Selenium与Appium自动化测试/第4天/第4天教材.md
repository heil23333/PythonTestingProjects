## 5. 先理解 Day 4 为什么要引入 pytest

到 Day 3 为止，你写的大多还是这种结构：

```python
driver = webdriver.Edge()
driver.get("...")
...
assert ...
driver.quit()
```

这种脚本有两个问题：

1. 浏览器启动和关闭逻辑会到处重复
2. 成功、失败、多组数据很难组织

所以 Day 4 要把它升级成“测试用例”。

你可以先把 `pytest` 理解成：

- 一个更适合跑测试的执行器
- 一个帮你组织测试生命周期的工具
- 一个能自然接入参数化、夹具和报告的框架

### 5.1 Day 4 第一个转变

不是去学一堆新语法，而是把原来的脚本换个组织方式。

例如，最小测试用例会长这样：

```python
def test_login_success(driver):
    driver.get("http://127.0.0.1:8000/pages/day4_login.html")
    driver.find_element(By.ID, "day4-username").send_keys("tester")
    driver.find_element(By.ID, "day4-password").send_keys("Selenium123")
    Select(driver.find_element(By.ID, "day4-role")).select_by_value("qa")
    driver.find_element(By.ID, "day4-login").click()

    wait = WebDriverWait(driver, 10)
    wait.until(EC.url_contains("day4_dashboard"))
    assert "欢迎你，tester" in driver.find_element(By.ID, "welcome-message").text
```

这里真正新增的只有两件事：

- 测试函数命名开始遵守 `test_`
- 浏览器实例不再手写在函数里，而是交给 `fixture`

### 5.1.1 这段代码到底在做什么

```python
def test_login_success(driver):
```

这行的意思不是“声明一个普通函数”，而是：

- 这是一个会被 `pytest` 自动发现的测试函数
- `driver` 不是你手动传进来的，而是 `pytest` 去找同名 `fixture` 注入给你的

```python
driver.get("http://127.0.0.1:8000/pages/day4_login.html")
```

这行是在做“前置动作”：

- 进入登录页
- 保证每条测试都从统一页面开始

```python
driver.find_element(By.ID, "day4-username").send_keys("tester")
driver.find_element(By.ID, "day4-password").send_keys("Selenium123")
Select(driver.find_element(By.ID, "day4-role")).select_by_value("qa")
```

这三行分别对应：

- 输入用户名
- 输入密码
- 选择角色

这里特别要注意：

- 角色是原生 `<select>`，所以要用 `Select`
- 不是所有“看起来像下拉框”的控件都这样写，但 Day 4 这个页面就是原生 `<select>`

```python
driver.find_element(By.ID, "day4-login").click()
```

这行对应提交登录表单。

```python
wait = WebDriverWait(driver, 10)
wait.until(EC.url_contains("day4_dashboard"))
```

这里不是为了“等页面好看一点”，而是因为点击登录后页面要：

- 请求接口
- 判断成功结果
- 再跳转到仪表盘页

所以你必须等 URL 真的发生变化。

```python
assert "欢迎你，tester" in driver.find_element(By.ID, "welcome-message").text
```

最后这句才是断言：

- 前面都只是“操作页面”
- 到这里才是在验证结果到底对不对

你可以把整段测试分成四层：

1. 打开页面
2. 操作页面
3. 等待页面状态稳定
4. 断言结果

### 5.2 随堂检查

1. Day 4 为什么不建议继续到处复制 `driver = webdriver.Edge()`？
2. Day 4 引入 `pytest` 的第一步，是先学复杂框架，还是先把脚本变成测试函数？

### 5.3 答案

1. 因为浏览器生命周期会大量重复，后面维护和扩展会越来越乱。
2. 先把脚本变成测试函数。

### 5.4 你提到的 pytest-selenium 插件是什么

有，而且确实存在。

它的核心思路是：

- 由插件直接给你提供一个 `selenium` fixture
- 你在测试函数里不再自己定义浏览器 `fixture`
- 运行时通过命令行参数指定浏览器

最小写法会更像这样：

```python
def test_login_success(selenium):
    selenium.get("http://127.0.0.1:8000/pages/day4_login.html")
    selenium.find_element(By.ID, "day4-username").send_keys("tester")
    selenium.find_element(By.ID, "day4-password").send_keys("Selenium123")
    Select(selenium.find_element(By.ID, "day4-role")).select_by_value("qa")
    selenium.find_element(By.ID, "day4-login").click()
```

运行时通常写成：

```bash
pytest --driver Edge
```

你可以先把它理解成：

`插件帮你把浏览器 fixture 这层先封起来了。`

### 5.5 Day 4 为什么主线先不教 pytest-selenium

不是它不好，而是它会把一个关键学习点“藏起来”：

- 浏览器是怎么创建的
- 生命周期是谁在管理
- 为什么测试函数里能直接拿到 driver

如果一开始就直接上插件，你很容易“会用，但不知道为什么这样用”。

所以 Day 4 主线刻意先教：

- 纯 `pytest + selenium`
- 自己写最小 `fixture`
- 自己理解 `yield`

等你把底层思路抓稳了，再回头用 `pytest-selenium`，你会更清楚：

- 它帮你省掉了什么
- 它适合什么场景
- 它什么时候反而不如自己写 `fixture` 灵活

## 6. fixture 是什么，为什么它要先于参数化

`fixture` 可以先理解成：

`给测试提前准备好它需要的东西。`

Day 4 最常见的就是给每个测试准备一个浏览器。

最小写法：

```python
import pytest
from selenium import webdriver


@pytest.fixture
def driver():
    browser = webdriver.Edge()
    browser.maximize_window()
    yield browser
    browser.quit()
```

这里的 `yield` 前后分别表示：

- 前面：准备资源
- 后面：清理资源

### 6.0.1 这段 fixture 要逐行怎么看

```python
@pytest.fixture
```

这行是在告诉 `pytest`：

- 下面这个函数不是普通工具函数
- 它是一个可被测试函数依赖的资源提供者

```python
def driver():
```

这里的函数名很重要，因为：

- 你的测试函数如果写 `def test_xxx(driver):`
- `pytest` 就会按名字去找这个 `driver` fixture

```python
browser = webdriver.Edge()
browser.maximize_window()
```

这两行是在做资源准备：

- 启动浏览器
- 可选地放大窗口，减少一些元素不可见问题

```python
yield browser
```

这是 Day 4 的核心句子。

它的含义是：

- `yield` 之前：先准备好浏览器
- 把这个浏览器交给测试函数去用
- 测试函数跑完后，再回来执行 `yield` 后面的清理逻辑

```python
browser.quit()
```

这行就是资源回收。

你可以把整段 `fixture` 理解成：

`先借你一个浏览器，用完我再负责回收。`

### 6.1 为什么 fixture 要先讲

因为你后面无论是：

- 参数化
- 失败截图
- POM

都默认要先有一个稳定的 `driver` 来源。

如果 `driver` 的启动和回收都还没讲清楚，后面内容会越来越乱。

### 6.2 scope 现在先怎么用

Day 4 建议先用默认的函数级 `fixture`。

也就是：

- 每个测试函数各用一个全新的浏览器
- 用例之间互不污染

先不要急着上 `scope="class"` 或 `scope="session"`，那是优化问题，不是 Day 4 主线。

### 6.3 随堂检查

1. `fixture` 在 Day 4 最常见的用途是什么？
2. `yield` 后面的代码通常负责什么？
3. Day 4 为什么建议先用函数级别的浏览器 `fixture`？

### 6.4 答案

1. 给测试准备浏览器、页面对象或测试数据。
2. 负责清理资源，例如关闭浏览器。
3. 因为隔离性更好，不同测试之间不会互相污染。

## 7. 参数化为什么放在 fixture 之后

参数化的核心作用是：

`同一条测试逻辑，跑多组数据。`

Day 4 登录页是最适合练参数化的场景。

比如：

```python
import pytest


@pytest.mark.parametrize(
    "username,password,role,expected_success",
    [
        ("tester", "Selenium123", "qa", True),
        ("tester", "wrong", "qa", False),
        ("someone", "Selenium123", "qa", False),
        ("tester", "Selenium123", "dev", False),
    ]
)
def test_login_cases(driver, username, password, role, expected_success):
    ...
```

### 7.0.1 参数化这段代码到底在发生什么

```python
@pytest.mark.parametrize(
    "username,password,role,expected_success",
    [
        ("tester", "Selenium123", "qa", True),
        ("tester", "wrong", "qa", False),
        ("someone", "Selenium123", "qa", False),
        ("tester", "Selenium123", "dev", False),
    ]
)
```

这段不是“定义一个列表”这么简单，而是在告诉 `pytest`：

- 把同一个测试函数拆成 4 次独立执行
- 每次分别喂一组登录数据

也就是说，下面这个函数：

```python
def test_login_cases(driver, username, password, role, expected_success):
```

看起来只写了一次，但运行时会变成 4 条测试。

这也是为什么参数化这么适合登录测试：

- 页面操作流程相同
- 只是账号组合不同

### 7.0.2 Day 4 里参数化测试最推荐怎么写

最稳的结构是：

1. 每次测试先重新打开登录页
2. 执行同一套登录动作
3. 根据 `expected_success` 分支断言

示意：

```python
def test_login_cases(driver, username, password, role, expected_success):
    driver.get("http://127.0.0.1:8000/pages/day4_login.html")

    driver.find_element(By.ID, "day4-username").clear()
    driver.find_element(By.ID, "day4-username").send_keys(username)
    driver.find_element(By.ID, "day4-password").clear()
    driver.find_element(By.ID, "day4-password").send_keys(password)
    Select(driver.find_element(By.ID, "day4-role")).select_by_value(role)
    driver.find_element(By.ID, "day4-login").click()

    wait = WebDriverWait(driver, 10)

    if expected_success:
        wait.until(EC.url_contains("day4_dashboard"))
        assert "欢迎你" in driver.find_element(By.ID, "welcome-message").text
        assert driver.find_element(By.ID, "role-badge").get_attribute("data-role") == "qa"
    else:
        wait.until(EC.text_to_be_present_in_element((By.ID, "login-feedback"), "登录失败"))
        assert "登录失败" in driver.find_element(By.ID, "login-feedback").text
        assert driver.current_url.endswith("day4_login.html")
```

这段你要重点看懂的是：

- 成功和失败不是共用一套断言
- 成功断言看跳转和欢迎语
- 失败断言看错误提示和当前 URL

### 7.1 Day 4 里怎么断言成功和失败

成功时，你可以断言：

- URL 跳到 `day4_dashboard.html`
- 欢迎语包含用户名
- `role-badge` 的 `data-role` 等于 `qa`

失败时，你可以断言：

- 还停留在登录页
- `login-feedback` 的文本等于错误提示
- `login-feedback` 的 `data-status` 等于 `error`

### 7.2 Day 4 参数化最容易写错的地方

1. 成功和失败断言写成一套，结果逻辑混乱
2. 每组数据前没有回到初始页
3. 测试函数里把数据、定位、断言全部揉在一起

### 7.3 随堂检查

1. 参数化最适合解决 Day 4 的哪类重复问题？
2. Day 4 登录测试里，成功分支和失败分支的断言应该完全一样吗？

### 7.4 答案

1. 同一条登录逻辑要反复跑不同账号组合的问题。
2. 不应该，成功和失败的页面结果本来就不同。

## 8. 失败截图放在 Day 4，为什么不是更早

因为到 Day 4 你才正式进入“测试用例”阶段。

脚本阶段也能截图，但只有进了测试框架后，截图才真正变成：

- 帮你排查失败
- 帮你保留证据
- 帮你和报告、日志一起使用

### 8.1 最简单的 Day 4 思路

先别急着学复杂 hook。

今天先学最朴素的思路：

```python
from pathlib import Path


def save_failure_screenshot(driver, name):
    screenshot_dir = Path("artifacts/screenshots")
    screenshot_dir.mkdir(parents=True, exist_ok=True)
    driver.save_screenshot(str(screenshot_dir / f"{name}.png"))
```

如果某个断言前你觉得很关键，可以先保存截图，再让断言失败。

### 8.2 Day 4 先达到什么标准就够了

先做到下面这件事就够：

- 测试失败时，你知道怎么手动留一张图

不要一上来就把截图、日志、报告、hook 全部揉在一起。

### 8.3 随堂检查

### 8.4 答案

## 9. 什么是 POM，为什么它必须放在后面讲

`POM` 是 `Page Object Model`，页面对象模型。

你可以先把它理解成：

`把页面上的定位和操作，收进页面类里。`

### 9.1 Day 4 为什么不能一开始就讲 POM

因为如果你还没搞清楚：

- 什么是测试函数
- 什么是 fixture
- 什么是参数化

那你写 `POM` 往往只会变成：

- 一堆类
- 一堆方法
- 但其实你不知道测试逻辑应该放哪里

所以 Day 4 正确顺序是：

1. 先能写测试
2. 再把重复页面操作抽出去

### 9.2 Day 4 的最小 POM 划分

登录页负责：

- 输入用户名
- 输入密码
- 选择角色
- 点击登录
- 读取错误提示

仪表盘页负责：

- 读取欢迎语
- 读取角色标签
- 判断退出按钮是否可见

示意代码：

```python
class LoginPage:
    def __init__(self, driver):
        self.driver = driver

    def open(self):
        self.driver.get("http://127.0.0.1:8000/pages/day4_login.html")

    def login(self, username, password, role):
        self.driver.find_element(By.ID, "day4-username").clear()
        self.driver.find_element(By.ID, "day4-username").send_keys(username)
        self.driver.find_element(By.ID, "day4-password").clear()
        self.driver.find_element(By.ID, "day4-password").send_keys(password)
        Select(self.driver.find_element(By.ID, "day4-role")).select_by_value(role)
        self.driver.find_element(By.ID, "day4-login").click()

    def feedback_text(self):
        return self.driver.find_element(By.ID, "login-feedback").text
```

```python
class DashboardPage:
    def __init__(self, driver):
        self.driver = driver

    def welcome_text(self):
        return self.driver.find_element(By.ID, "welcome-message").text

    def role_value(self):
        return self.driver.find_element(By.ID, "role-badge").get_attribute("data-role")
```

### 9.2.1 LoginPage 这段代码为什么这么拆

```python
class LoginPage:
```

这不是为了“显得高级”，而是为了把“登录页相关操作”收拢到一起。

```python
def open(self):
    self.driver.get("http://127.0.0.1:8000/pages/day4_login.html")
```

这行表示：

- 打开页面的动作放在页面类里
- 测试文件不需要到处写 URL

```python
def login(self, username, password, role):
```

这里不是只支持一组固定数据，而是把数据参数化：

- 用户名可以变
- 密码可以变
- 角色可以变

这样参数化测试就可以直接复用这个方法。

```python
self.driver.find_element(By.ID, "day4-username").clear()
...
self.driver.find_element(By.ID, "day4-login").click()
```

这块的本质是：

- 页面对象负责“怎么操作页面”
- 测试函数负责“用哪组数据调用它”

```python
def feedback_text(self):
    return self.driver.find_element(By.ID, "login-feedback").text
```

这个方法不是断言，它只是“取结果”。

Day 4 这里很关键：

- 页面对象负责取值
- 测试函数负责断言

### 9.2.2 DashboardPage 为什么只保留读取方法

因为 Day 4 这个仪表盘页本身动作不多，它更适合承担：

- 读取欢迎语
- 读取角色标签
- 读取按钮可见状态

所以像这样：

```python
def welcome_text(self):
    return self.driver.find_element(By.ID, "welcome-message").text
```

本质上是在帮你把定位细节藏起来。

测试文件以后就可以写成：

```python
assert "tester" in dashboard.welcome_text()
assert dashboard.role_value() == "qa"
```

这样测试会更像“业务表达”，而不是一堆裸 `find_element`。

### 9.2.3 POM 化之后，测试用例到底该怎么写

这是 Day 4 很关键的一跳。

很多人把页面对象写出来以后，测试文件反而不会写了，常见原因是：

- 不知道页面对象和测试函数谁负责什么
- 不知道成功和失败分支该放哪一层
- 不知道测试里还要不要保留等待和断言

你先记一个最简单的原则：

`POM 化之后，测试函数负责“组织场景”，页面对象负责“操作页面”。`

也就是说，测试函数通常只做这 4 件事：

1. 准备页面对象
2. 调用页面动作
3. 等待业务结果成立
4. 做业务断言

#### 9.2.3.1 没做 POM 时，测试常常长这样

```python
def test_login_success(driver):
    driver.get("http://127.0.0.1:8000/pages/day4_login.html")
    driver.find_element(By.ID, "day4-username").send_keys("tester")
    driver.find_element(By.ID, "day4-password").send_keys("Selenium123")
    Select(driver.find_element(By.ID, "day4-role")).select_by_value("qa")
    driver.find_element(By.ID, "day4-login").click()

    wait = WebDriverWait(driver, 10)
    wait.until(EC.url_contains("day4_dashboard"))
    assert "欢迎你，tester" in driver.find_element(By.ID, "welcome-message").text
    assert driver.find_element(By.ID, "role-badge").get_attribute("data-role") == "qa"
```

这段能跑，但问题是：

- 定位都写在测试里
- 登录动作以后会反复复制
- 页面一多，测试文件会越来越臃肿

#### 9.2.3.2 做了 POM 之后，测试应该变成这样

```python
def test_login_success(driver):
    login_page = LoginPage(driver)
    dashboard_page = DashboardPage(driver)

    login_page.open()
    login_page.login("tester", "Selenium123", "qa")

    wait = WebDriverWait(driver, 10)
    wait.until(EC.url_contains("day4_dashboard"))

    assert "欢迎你，tester" in dashboard_page.welcome_text()
    assert dashboard_page.role_value() == "qa"
```

你会发现这里的测试文件明显变短了，但更重要的是语义更清楚了：

- `login_page.open()`：打开登录页
- `login_page.login(...)`：执行登录动作
- `dashboard_page.welcome_text()`：读取成功后的欢迎语
- `dashboard_page.role_value()`：读取成功后的角色标签

这就是 POM 最核心的收益：

`测试文件开始更像“业务步骤”，而不是一堆元素定位细节。`

#### 9.2.3.3 这段 POM 测试该逐行怎么理解

```python
login_page = LoginPage(driver)
dashboard_page = DashboardPage(driver)
```

这里不是“页面对象自己创建浏览器”，而是：

- 浏览器仍然来自 `fixture`
- 页面对象只是拿这个浏览器去操作不同页面

```python
login_page.open()
login_page.login("tester", "Selenium123", "qa")
```

这里表示：

- “怎么打开登录页”
- “怎么输入账号并点击登录”

这些页面细节现在都被收进了 `LoginPage`。

```python
wait.until(EC.url_contains("day4_dashboard"))
```

这句仍然放在测试层，而不是页面对象里，原因是：

- 它更像“业务结果是否成立”
- 而不是“页面动作本身怎么执行”

```python
assert "欢迎你，tester" in dashboard_page.welcome_text()
assert dashboard_page.role_value() == "qa"
```

这两句是最典型的 POM 后测试写法：

- 页面对象负责“给我值”
- 测试函数负责“判断值对不对”

#### 9.2.3.4 参数化后，POM 测试用例怎么写

很多人写到这一步又会卡住：如果已经有 `LoginPage` 了，参数化测试是不是也要改？

答案是：要改，但改法其实很自然。

```python
@pytest.mark.parametrize(
    "username,password,role,expected_success",
    [
        ("tester", "Selenium123", "qa", True),
        ("tester", "wrong", "qa", False),
        ("someone", "Selenium123", "qa", False),
        ("tester", "Selenium123", "dev", False),
    ]
)
def test_login_cases(driver, username, password, role, expected_success):
    login_page = LoginPage(driver)
    dashboard_page = DashboardPage(driver)

    login_page.open()
    login_page.login(username, password, role)

    wait = WebDriverWait(driver, 10)

    if expected_success:
        wait.until(EC.url_contains("day4_dashboard"))
        assert "欢迎你" in dashboard_page.welcome_text()
        assert dashboard_page.role_value() == "qa"
    else:
        wait.until(
            EC.text_to_be_present_in_element((By.ID, "login-feedback"), "登录失败")
        )
        assert "登录失败" in login_page.feedback_text()
```

你要重点看清楚两件事：

第一，参数化以后，页面对象没有变复杂。

变的是测试函数喂给它的数据：

- 成功账号
- 错误密码
- 错误用户名
- 错误角色

第二，成功和失败仍然不是同一套断言。

成功时看：

- URL 跳转
- 欢迎语
- 角色标签

失败时看：

- 错误提示
- 当前仍停留在登录页

#### 9.2.3.5 POM 化之后，测试文件不要做什么

测试文件里不要再把这些东西重新写回去：

- 大量重复的 `find_element`
- 登录页元素定位细节
- 仪表盘页元素定位细节

否则你虽然建了 `LoginPage` 和 `DashboardPage`，但其实没有真正用上 POM。

#### 9.2.3.6 一句总结

`POM 化之后，测试文件写流程，页面对象写细节。`

### 9.3 Day 4 里哪些内容不要急着塞进 POM

先不要把这些东西都硬塞进去：

- 所有断言
- 所有等待
- 所有业务分支
- 所有工具函数

Day 4 最小目标只是：

- 页面定位更集中
- 页面动作更清晰

### 9.4 随堂检查

1. `POM` 在 Day 4 的核心价值是什么？
2. Day 4 里，`LoginPage` 应该更偏向装测试断言，还是装页面操作？

### 9.5 答案

1. 把页面定位和页面操作集中管理，降低重复和散乱。
2. 更偏向装页面操作。

### 10.1 一套最小可运行骨架该怎么拼起来

很多人看到 `conftest.py`、`pages/`、`tests/` 会有点散，所以这里把最小关系串一下。

`conftest.py`：

```python
import pytest
from selenium import webdriver


@pytest.fixture
def driver():
    browser = webdriver.Edge()
    yield browser
    browser.quit()
```

`pages/login_page.py`：

```python
from selenium.webdriver.common.by import By
from selenium.webdriver.support.select import Select


class LoginPage:
    def __init__(self, driver):
        self.driver = driver

    def open(self):
        self.driver.get("http://127.0.0.1:8000/pages/day4_login.html")

    def login(self, username, password, role):
        self.driver.find_element(By.ID, "day4-username").clear()
        self.driver.find_element(By.ID, "day4-username").send_keys(username)
        self.driver.find_element(By.ID, "day4-password").clear()
        self.driver.find_element(By.ID, "day4-password").send_keys(password)
        Select(self.driver.find_element(By.ID, "day4-role")).select_by_value(role)
        self.driver.find_element(By.ID, "day4-login").click()

    def feedback_text(self):
        return self.driver.find_element(By.ID, "login-feedback").text
```

`tests/test_login.py`：

```python
import pytest
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC
from pages.login_page import LoginPage
from pages.dashboard_page import DashboardPage


@pytest.mark.parametrize(
    "username,password,role,ok",
    [
        ("tester", "Selenium123", "qa", True),
        ("tester", "wrong", "qa", False),
    ]
)
def test_login(driver, username, password, role, ok):
    login_page = LoginPage(driver)
    dashboard_page = DashboardPage(driver)

    login_page.open()
    login_page.login(username, password, role)

    wait = WebDriverWait(driver, 10)
    if ok:
        wait.until(EC.url_contains("day4_dashboard"))
        assert "欢迎你" in dashboard_page.welcome_text()
        assert dashboard_page.role_value() == "qa"
    else:
        wait.until(EC.text_to_be_present_in_element((By.ID, "login-feedback"), "登录失败"))
        assert "登录失败" in login_page.feedback_text()
```

你要看懂它们之间的关系：

- `conftest.py` 提供浏览器
- `pages/` 提供页面动作和页面读取
- `tests/` 负责组合流程、等待结果和断言

这三层一旦分清，Day 4 就不会再乱。

### 10.1 Day 4 不建议的做法

- `common.py` 里塞所有东西
- 还没写几个用例，就先抽十几层基类

### 14.2 代码理解题

看下面代码，回答问题：

```python

    browser.quit()

```

请回答：

1. `driver` 为什么放在 `fixture` 里，而不是每个测试里自己创建？
2. `parametrize` 在这里解决了什么问题？
3. 如果登录成功，Day 4 最适合断言哪几类结果？

### 14.3 标准答案

1. 因为浏览器生命周期是公共逻辑，重复写会增加维护成本。

### 14.4 概念题参考答案

1. 因为重复太多，维护成本高，也不利于扩展多组数据和统一清理。
2. 最常见是负责准备和回收浏览器。
3. 因为 `POM` 依赖你先有稳定的测试执行结构，否则只会变成形式上的封装。
