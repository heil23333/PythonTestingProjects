## 4. 重点一：先把脚本变成 pytest 用例

### 4.1 一句话理解

Day 4 最先做的，不是建复杂框架，而是把“脚本”改成“测试函数”。

### 4.2 最小变化

- 函数名变成 `test_`
- `assert` 保留下来
- 浏览器不再写死在每个测试里

### 4.2.1 关键代码怎么读

如果你看到：

```python
def test_login_success(driver):
```

这里其实同时发生了两件事：

- `test_` 让它变成 `pytest` 测试函数
- `driver` 让它依赖一个同名 `fixture`

如果你看到：

```python
wait.until(EC.url_contains("day4_dashboard"))
```

你就要立刻想到：

- 这里等的是“登录成功后的页面跳转”
- 不是随便加个等待

如果你看到：

```python
assert "欢迎你，tester" in ...
```

这里就是在做结果断言，而不是页面操作。

### 4.3 随堂检查

1. Day 4 的第一步，是先学 `POM`，还是先把脚本改成 `pytest` 测试函数？
2. 为什么 Day 4 不适合继续把登录流程写成一个个独立脚本？

### 4.4 答案

1. 先把脚本改成 `pytest` 测试函数。
2. 因为重复逻辑会越来越多，后面参数化和管理浏览器都很不方便。

## 5. 重点二：fixture

### 5.1 一句话理解

`fixture` 就是提前给测试准备资源。

### 5.2 Day 4 最典型的 fixture

- 浏览器启动
- 浏览器关闭

### 5.3 最小示意

```python
@pytest.fixture
def driver():
    browser = webdriver.Edge()
    yield browser
    browser.quit()
```

你要重点记住：

- `@pytest.fixture`：把函数变成资源提供者
- `yield browser`：把浏览器借给测试函数
- `browser.quit()`：测试跑完后收回浏览器

### 5.4 随堂检查

1. `yield` 前后分别通常负责什么？
2. Day 4 为什么建议先用函数级别浏览器 fixture？

### 5.5 答案

1. 前面负责准备资源，后面负责清理资源。
2. 因为隔离性更好，不容易互相污染。

## 6. 重点三：参数化

### 6.1 一句话理解

同一条登录逻辑，跑多组数据。

### 6.2 Day 4 最适合的参数化场景

- 正确用户名 + 正确密码 + 正确角色
- 正确用户名 + 错误密码
- 错误用户名
- 正确用户名 + 正确密码 + 错误角色

最容易看懂的参数化心智模型是：

- 你不是写了 1 条测试
- 你是写了 1 个“测试模板”
- `pytest` 再拿多组数据去把它展开

### 6.3 随堂检查

1. 参数化最适合解决 Day 4 的哪类重复？
2. 成功登录和失败登录的断言应该写成完全一样吗？

### 6.4 答案

1. 同一条登录逻辑要跑多组账号组合的重复。
2. 不应该，页面结果本来就不同。

## 7. 重点四：失败截图

### 7.1 一句话理解

截图不是为了“好看”，而是为了失败时能快速定位现场。

### 7.2 Day 4 先做到什么就够

- 知道怎么手动保存截图
- 知道截图应该落到 `artifacts/screenshots/`

最小函数可以先这样记：

```python
from pathlib import Path


def save_failure_screenshot(driver, name):
    folder = Path("artifacts/screenshots")
    folder.mkdir(parents=True, exist_ok=True)
    driver.save_screenshot(str(folder / f"{name}.png"))
```

这里不要只记“会抄”，要看懂：

- `Path(...)` 在准备目录
- `mkdir(..., exist_ok=True)` 在确保目录存在
- `save_screenshot(...)` 才是真正保存图片

### 7.3 随堂检查

1. Day 4 是否必须一开始就写复杂 hook 来做截图？
2. Day 4 为什么到这时才开始强调截图？

### 7.4 答案

1. 不必须，先会最简单的截图方式就够了。
2. 因为从 Day 4 开始你已经进入测试用例阶段，失败排查成本更高。

## 8. 重点五：POM

### 8.1 一句话理解

`POM` 是把页面定位和页面操作收进页面类里。

### 8.2 Day 4 的最小 POM 划分

- `LoginPage`
  - 打开登录页
  - 输入账号信息
  - 点击登录
  - 读取错误提示

- `DashboardPage`
  - 读取欢迎语
  - 读取角色
  - 判断退出按钮是否可见

这里的关键不是“会写 class”，而是分清边界：

- 页面对象：负责定位和动作
- 测试函数：负责场景组合和断言

如果你把这两层混了，Day 4 后面会越来越乱。

### 8.2.2 POM 化后，测试函数最小该长什么样

这是 Day 4 很容易卡住的地方。

页面对象写出来后，测试函数不要再退回去写一堆裸定位，而是应该像这样：

```python
def test_login_success(driver):
    login_page = LoginPage(driver)
    dashboard_page = DashboardPage(driver)

    login_page.open()
    login_page.login("tester", "Selenium123", "qa")

    wait = WebDriverWait(driver, 10)
    wait.until(EC.url_contains("day4_dashboard"))

    assert "欢迎你，tester" in dashboard_page.welcome_text()
    assert dashboard_page.role_value() == "qa"
```

你先记 4 步就够：

1. 创建页面对象
2. 调用页面动作
3. 等待业务结果
4. 做业务断言

一句最关键的话：

`POM 化之后，测试函数写流程，页面对象写细节。`

### 8.2.3 参数化之后也一样

参数化不会改变这个结构，只会改变“喂给页面对象的数据”。

也就是说：

- `login_page.login(username, password, role)` 还是那一行
- 变的是每次传进去的账号组合
- 成功和失败的断言仍然分开写

### 8.2.4 你提到的 pytest-selenium 插件

有，而且很常见。

它的典型写法是：

```python
def test_login_success(selenium):
    selenium.get("http://127.0.0.1:8000/pages/day4_login.html")
```

也就是说：

- 插件直接给你一个 `selenium` fixture
- 你运行时通常写 `pytest --driver Edge`

但 Day 4 主线先不直接教它，是因为：

- 你要先看懂 `fixture` 的本质
- 不然你会“能用插件”，但不清楚浏览器是谁创建和回收的

### 8.3 随堂检查

1. Day 4 里，`POM` 的核心价值是什么？
2. `LoginPage` 更应该偏向放页面动作，还是放完整测试逻辑？
3. POM 化之后，测试函数最核心的职责是什么？

### 8.4 答案

1. 把页面定位和动作集中管理，减少重复和散乱。
2. 更应该偏向放页面动作。
3. 负责组合场景、等待业务结果并做断言。

1. 创建 `LoginPage`

3. 把页面动作和读取方法封装进去

1. 故意让一组数据失败
2. 保存截图
3. 检查截图是否停留在登录页

### 9.5 参考答案方向

基础练习方向：

1. 测试函数里打开登录页
2. 填写有效账号
3. 点击登录
4. 等待进入仪表盘
5. 断言欢迎语

1. 用参数化定义账号组合
2. 每组数据都重新打开登录页
3. 成功断言看 URL、欢迎语、角色
4. 失败断言看错误提示

结构化练习方向：

1. 页面对象负责动作和读取
2. 测试文件负责组合调用、等待和断言
3. `fixture` 负责浏览器生命周期

## 11. 今天最容易混淆的几件事

- `fixture` 不是页面对象，它是资源准备机制
- 参数化不是封装，它是多组数据复用
- `POM` 不是把所有断言都塞进页面类
- 截图不是一定要 hook 才能做

这四件事只要分清，Day 4 就会顺很多。
