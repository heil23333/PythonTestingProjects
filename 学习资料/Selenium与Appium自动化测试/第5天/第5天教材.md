## 5. Appium 到底是什么

你可以先把 Appium 理解成：

`移动端版的 WebDriver 自动化中间层。`

它的作用不是自己去“看见”页面，而是：

- 接收你的 Python 代码命令
- 转给对应平台驱动
- 再由驱动去操作 Android 或 iOS 设备

所以 Appium 和 Selenium 很像，不是巧合。

它们共享了很大一部分自动化思想：

- 会话
- 定位
- 点击
- 输入
- 读取文本
- 等待
- 断言

区别在于：

- Selenium 主要面向浏览器
- Appium 主要面向移动端 App

### 5.1 Appium 的三层结构

你现在只要记住这 3 层：

1. 客户端
2. 服务端
3. 平台驱动

#### 客户端

就是你写的 Python 代码，例如：

```python
from appium import webdriver
```

#### 服务端

就是你终端里启动的：

```bash
appium
```

#### 平台驱动

不同平台用不同驱动：

- Android：`UiAutomator2`
- iOS：`XCUITest`

### 5.2 这三层是怎么连起来的

可以把流程理解成：

1. Python 代码发命令
2. Appium 服务接收命令
3. 服务把命令交给对应驱动
4. 驱动操作真机或模拟器
5. 再把结果返回给你

### 5.3 随堂检查

1. Appium 客户端是不是直接操控手机？
2. Android 和 iOS 可以共用一个平台驱动吗？
3. Appium 服务启动了，是否就代表已经能测 Android/iOS？

### 5.4 答案

1. 不是，它是通过 Appium 服务和平台驱动间接操作设备。
2. 不可以，Android 常用 `UiAutomator2`，iOS 常用 `XCUITest`。
3. 不代表，还要看对应驱动有没有安装、设备有没有连通。

## 6. 为什么 Appium 和 Selenium 长得这么像

这是 Day 5 一个非常值得建立的理解。

Appium Python 客户端本质上是在 Selenium WebDriver 的基础上扩展出来的。

所以你会看到这些熟悉的东西：

- `webdriver.Remote(...)`
- `find_element(...)`
- `WebDriverWait`
- `expected_conditions`
- `click()`
- `send_keys()`

也就是说：

你前 4 天学的很多东西不会作废，而是继续复用。

### 6.1 复用关系怎么理解

Web 自动化里你写：

```python
driver.find_element(By.ID, "username").send_keys("tester")
```

移动端里你写：

```python
driver.find_element(AppiumBy.ACCESSIBILITY_ID, "Views").click()
```

看起来不同，但思路完全一样：

- 先建立 driver 会话
- 再定位元素
- 再执行动作

### 6.2 Day 5 真正新增的东西

不是“自动化思维”，而是这些移动端专属内容：

- 平台驱动
- 设备连接
- 能力参数
- 移动端定位策略
- 原生 App 生命周期

## 7. 会话和能力参数是什么

这一块是 Appium 成功与否的第一道门。

你可以先把能力参数理解成：

`告诉 Appium：我要连哪种设备，用哪个驱动，打开哪个 App。`

Day 5 最推荐用新版 `Options` 写法，而不是一上来就手写旧式 `desired_capabilities` 字典。

### 7.1 Android 最小写法

```python
from appium.options.android import UiAutomator2Options

options = UiAutomator2Options()
options.platform_name = "Android"
options.automation_name = "UiAutomator2"
options.device_name = "Android Emulator"
options.app_package = "io.appium.android.apis"
options.app_activity = ".ApiDemos"
```

### 7.2 这几行分别在说什么

```python
options.platform_name = "Android"
```

告诉 Appium：目标平台是 Android。

```python
options.automation_name = "UiAutomator2"
```

告诉 Appium：Android 这边用 `UiAutomator2` 驱动。

```python
options.device_name = "Android Emulator"
```

这是设备名。学习阶段你可以把它理解成“给会话一个目标设备标识”。

```python
options.app_package = "io.appium.android.apis"
options.app_activity = ".ApiDemos"
```

这两行是在告诉 Android：

- 启动哪个包
- 进入哪个 Activity

这和 Web 打开 URL 有点像，只是移动端不是 URL，而是 App 入口。

### 7.3 iOS 对应写法

```python
from appium.options.ios import XCUITestOptions

options = XCUITestOptions()
options.platform_name = "iOS"
options.automation_name = "XCUITest"
options.device_name = "iPhone 16"
options.platform_version = "18.0"
options.bundle_id = "com.apple.Preferences"
```

你现在不用死记每个字段，只要理解：

- Android 更常见 `app_package + app_activity`
- iOS 更常见 `bundle_id` 或 `app`

### 7.4 随堂检查

1. 能力参数最核心是在描述什么？
2. Android 常见是通过什么来指定目标 App？
3. Day 5 为什么更推荐 `Options` 写法？

### 7.5 答案

1. 描述目标平台、驱动、设备和要启动的 App。
2. 常见是 `app_package` 和 `app_activity`。
3. 因为更清晰、更新，也更贴近当前客户端推荐方式。

## 8. 第一个 Android Appium 脚本怎么写

这部分是 Day 5 的主线实操。

### 8.1 最小可运行版本

```python
from appium import webdriver
from appium.options.android import UiAutomator2Options
from appium.webdriver.common.appiumby import AppiumBy
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

options = UiAutomator2Options()
options.platform_name = "Android"
options.automation_name = "UiAutomator2"
options.device_name = "Android Emulator"
options.app_package = "io.appium.android.apis"
options.app_activity = ".ApiDemos"

driver = webdriver.Remote("http://127.0.0.1:4723", options=options)

wait = WebDriverWait(driver, 10)
views = wait.until(
    EC.presence_of_element_located((AppiumBy.ACCESSIBILITY_ID, "Views"))
)
views.click()

animation = wait.until(
    EC.presence_of_element_located((AppiumBy.ACCESSIBILITY_ID, "Animation"))
)
assert animation.is_displayed()

driver.quit()
```

### 8.2 这段代码要逐行怎么看

```python
from appium import webdriver
```

这里不是 Selenium 的 `webdriver`，而是 Appium 的。

```python
from appium.options.android import UiAutomator2Options
```

这表示我们当前写的是 Android 会话，而且走 `UiAutomator2`。

```python
from appium.webdriver.common.appiumby import AppiumBy
```

移动端定位推荐用 `AppiumBy`，因为它支持更多移动端专属定位方式。

```python
driver = webdriver.Remote("http://127.0.0.1:4723", options=options)
```

这句和 Selenium 很像，但关键区别是：

- Selenium 连浏览器驱动
- Appium 连移动端自动化服务

```python
views = wait.until(
    EC.presence_of_element_located((AppiumBy.ACCESSIBILITY_ID, "Views"))
)
```

这里在做两件事：

- 等待页面加载出 `Views`
- 用 `ACCESSIBILITY_ID` 定位它

```python
views.click()
```

点击进入下一级菜单。

```python
animation = wait.until(
    EC.presence_of_element_located((AppiumBy.ACCESSIBILITY_ID, "Animation"))
)
assert animation.is_displayed()
```

这里的断言逻辑和 Selenium 是一样的：

- 先等目标元素出现
- 再断言它真的可见

### 8.3 这段代码说明了什么

它说明：

- Appium 和 Selenium 的主干写法非常像
- 你真正要适应的是会话配置和移动端定位方式

## 9. 移动端定位怎么选

Day 5 先掌握最常用的 3 类就够了。

### 9.1 `ACCESSIBILITY_ID`

这是 Day 5 最推荐优先尝试的定位。

```python
driver.find_element(AppiumBy.ACCESSIBILITY_ID, "Views")
```

优点：

- 语义清楚
- 跨平台相对更统一
- 在 Android / iOS 上都很常见

### 9.2 `ID`

Android 常见会遇到资源 id。

```python
driver.find_element(AppiumBy.ID, "com.demo:id/login_button")
```

这个定位也很稳，尤其在 Android 项目里经常很好用。

### 9.3 `XPATH`

只有前两种都不好用时，再考虑它。

```python
driver.find_element(AppiumBy.XPATH, "//android.widget.TextView[@text='Views']")
```

原因和 Web 很像：

- 太长会脆
- 太依赖层级
- 页面结构一改就容易坏

### 9.4 Android / iOS 的平台差异先怎么记

Android 常见：

- `ACCESSIBILITY_ID`
- `ID`
- `ANDROID_UIAUTOMATOR`

iOS 常见：

- `ACCESSIBILITY_ID`
- `ID` / `name`
- `IOS_PREDICATE`
- `IOS_CLASS_CHAIN`

Day 5 先不用全背，先抓住这句：

`优先 accessibility id，其次稳定 id，最后才是 xpath。`

### 9.5 随堂检查

1. Day 5 最推荐优先尝试哪种移动端定位？
2. Android 和 iOS 的定位方式是否完全一样？
3. 为什么 XPath 在移动端也不应该默认优先？

### 9.6 答案

1. `ACCESSIBILITY_ID`
2. 不完全一样，但有一部分策略是共通的。
3. 因为它也容易依赖结构、过长且不稳定。

## 11. Day 5 里的等待机制

Appium 里等待思想和 Selenium 完全共通。

最常用还是：

```python
wait = WebDriverWait(driver, 10)
wait.until(...)
```

### 11.1 为什么 Appium 也必须用等待

因为移动端也有大量动态过程：

- App 启动耗时
- 页面切换
- 动画
- 接口返回
- 软键盘弹出

所以你也不能乱用 `sleep()`。

### 11.2 Day 5 最常用等待

```python
EC.presence_of_element_located(...)
EC.visibility_of_element_located(...)
EC.element_to_be_clickable(...)
```

你前面学过的等待思路，今天直接继续用。

- 环境链路通常更短
- 很多定位和启动参数更直观

但你要同步记住 iOS 的对应关系：

- Android 常见入口：`app_package + app_activity`
- iOS 常见入口：`bundle_id` 或 `app`

- Android 常见平台特有定位：`ANDROID_UIAUTOMATOR`
- iOS 常见平台特有定位：`IOS_PREDICATE`、`IOS_CLASS_CHAIN`
